# 任务视频上传接口对接文档

## 1. 适用范围

本文档面向机器人端、边缘端或其它客户端开发者，用于对接媒体服务的任务视频上传接口。

适用场景：

1. 巡检、巡逻、安防等任务执行完成后上传任务视频。
2. 一个任务产生多个视频文件，逐个上传。
3. 上传过程中需要支持大文件、断点续传、并发分片和失败重试。
4. 上传完成后，媒体服务继续处理视频，生成可播放的 HLS 资源。

本文档只描述上传协议，不要求调用方使用仓库内现有 Python 或 Go 客户端代码。

## 2. 基础地址

媒体服务地址由部署环境决定：

```text
{MEDIA_SERVICE_URL}
```

示例：

```text
http://175.155.35.79:8088
```

任务视频上传接口统一前缀：

```text
{MEDIA_SERVICE_URL}/api/media/files
```

如果媒体服务 8088 是 HTTP 端口，请使用：

```text
http://host:8088
```

不要使用：

```text
https://host:8088
```

## 3. 请求头

媒体服务 JSON 接口建议携带：

```http
Content-Type: application/json
X-Robot-Id: {robotId}
```

说明：

| Header | 是否必填 | 说明 |
|---|---|---|
| `Content-Type` | 是 | JSON 接口固定 `application/json` |
| `X-Robot-Id` | 建议必填 | 机器人 ID；部分接口会用它做机器人维度校验 |

`PUT uploadUrl` 不需要 `X-Robot-Id`，因为 `uploadUrl` 是对象存储预签名 URL，鉴权信息已经包含在 URL 中。

## 4. 总体流程

```text
本地任务视频生成完成
  -> 生成稳定 sourceFileId
  -> POST /multipart-uploads 创建或恢复上传
  -> 根据 uploadedParts 判断缺失分片
  -> 使用首批 partUrls 或调用 /part-urls 获取上传地址
  -> PUT uploadUrl 上传每个缺失分片
  -> 所有分片 PUT 成功
  -> POST /complete 通知媒体服务完成上传
  -> 视频进入 PROCESSING
  -> GET /{fileId}/status 轮询
  -> READY 后可播放/下载
```

重点：

1. `complete` 成功只代表原始视频文件上传并合并完成。
2. 视频能否播放还要等媒体服务 HLS 后处理完成。
3. 客户端必须保存 `fileId`，后续播放、下载、任务产物查询都会用到。
4. 客户端必须保存 `sourceFileId`、`uploadId`、上传状态，便于断点续传。

## 5. sourceFileId 规则

`sourceFileId` 是客户端源文件的稳定幂等 ID，是断点续传的关键。

必须满足：

1. 同一个本地视频重试上传时，`sourceFileId` 必须相同。
2. 不同视频文件不能共用同一个 `sourceFileId`。
3. 不要每次重试都随机生成 `sourceFileId`。

推荐格式：

```text
{robotId}/{taskExecutionId或localTaskId}/{deviceId}/{fileName}/{fileSize}/{mtime}
```

示例：

```text
test001/task-exec-20260722-001/camera01/visible_20260722_152451.mp4/18268330/1784705200
```

如果任务执行记录 ID 还没有生成，可以使用本地任务批次 ID：

```text
test001/local-task-20260722-1524/camera01/visible_20260722_152451.mp4/18268330/1784705200
```

不要只使用：

```text
visible_20260722_152451.mp4
```

原因是同名文件很常见，且无法稳定支持断点续传。

## 6. metadata 建议

`metadata` 是字符串字段，建议传 JSON 字符串。

任务视频推荐内容：

```json
{
  "type": "task_recording",
  "cameraId": "camera01",
  "channel": "visible",
  "taskExecutionId": "task-exec-20260722-001",
  "localTaskId": "local-task-20260722-1524",
  "recordingStartedAt": "2026-07-22T15:24:51+08:00",
  "recordingEndedAt": "2026-07-22T15:27:12+08:00"
}
```

字段说明：

| 字段 | 建议 | 说明 |
|---|---|---|
| `type` | 必传 | 任务视频建议固定 `task_recording` |
| `cameraId` | 必传 | 摄像头或设备 ID |
| `channel` | 建议 | 如 `visible`、`thermal` |
| `taskExecutionId` | 有则传 | 任务执行记录 ID |
| `localTaskId` | 建议 | 无任务执行记录 ID 时的本地临时任务 ID |
| `recordingStartedAt` | 建议 | 视频开始时间 |
| `recordingEndedAt` | 建议 | 视频结束时间 |

注意：`metadata` 请求字段本身是字符串，不是 JSON 对象。

## 7. 接口一：创建或恢复分片上传

### 7.1 请求

```http
POST {MEDIA_SERVICE_URL}/api/media/files/multipart-uploads
Content-Type: application/json
X-Robot-Id: test001
```

请求体：

```json
{
  "robotId": "test001",
  "deviceId": "camera01",
  "extensionId": null,
  "sourceFileId": "test001/task-exec-20260722-001/camera01/visible_20260722_152451.mp4/18268330/1784705200",
  "fileType": "VIDEO",
  "fileName": "visible_20260722_152451.mp4",
  "contentType": "video/mp4",
  "fileSize": 18268330,
  "metadata": "{\"type\":\"task_recording\",\"cameraId\":\"camera01\",\"channel\":\"visible\",\"taskExecutionId\":\"task-exec-20260722-001\"}"
}
```

字段说明：

| 字段 | 是否必填 | 说明 |
|---|---|---|
| `robotId` | 建议必填 | 机器人 ID；如果 body 和 `X-Robot-Id` 都有，body 优先 |
| `deviceId` | 建议必填 | 摄像头或设备 ID |
| `extensionId` | 否 | 扩展绑定 ID，没有可传 `null` |
| `sourceFileId` | 强烈建议必填 | 本地源文件幂等 ID |
| `fileType` | 是 | 任务视频固定 `VIDEO` |
| `fileName` | 是 | 原始视频文件名 |
| `contentType` | 是 | MP4 使用 `video/mp4` |
| `fileSize` | 是 | 文件大小，单位字节，必须大于 0 |
| `metadata` | 建议 | JSON 字符串 |

### 7.2 响应

```json
{
  "fileId": "file_abc123",
  "uploadId": "upl_abc123",
  "uploadMode": "MULTIPART",
  "status": "UPLOADING",
  "partSize": 16777216,
  "partCount": 2,
  "uploadedParts": [
    {
      "partNumber": 1,
      "etag": "etag-value",
      "size": 16777216
    }
  ],
  "partUrls": [
    {
      "partNumber": 2,
      "uploadUrl": "http://minio-presigned-url-for-part-2"
    }
  ],
  "expiresAt": "2026-07-25T15:26:17+08:00"
}
```

字段说明：

| 字段 | 说明 |
|---|---|
| `fileId` | 媒体文件 ID，必须保存 |
| `uploadId` | 上传会话 ID；非空表示需要继续上传分片 |
| `uploadMode` | 上传模式，分片上传为 `MULTIPART` |
| `status` | 文件状态 |
| `partSize` | 每个分片大小，最后一片可能小于该值 |
| `partCount` | 分片总数 |
| `uploadedParts` | 服务端已经确认存在的分片，客户端应跳过 |
| `partUrls` | 服务端返回的首批分片上传 URL |
| `expiresAt` | 上传会话过期时间 |

### 7.3 响应处理

| 情况 | 客户端处理 |
|---|---|
| `uploadId` 非空 | 上传缺失分片 |
| `uploadId` 为空且 `status=PROCESSING` | 原始文件已上传完成，开始轮询状态 |
| `uploadId` 为空且 `status=READY` | 文件已可用，不需要再传 |
| `uploadedParts` 非空 | 跳过这些 part |
| `partUrls` 非空 | 可直接使用这些 URL 上传 |
| `partUrls` 不足或过期 | 调用 `/part-urls` 重新获取 |

## 8. 接口二：上传分片

### 8.1 分片计算

分片编号从 1 开始：

```text
offset = (partNumber - 1) * partSize
size = min(partSize, fileSize - offset)
```

示例：

```text
fileSize = 18268330
partSize = 16777216
partCount = 2

part 1:
  offset = 0
  size = 16777216

part 2:
  offset = 16777216
  size = 1491114
```

### 8.2 PUT 上传

```http
PUT {uploadUrl}
Content-Length: {size}

{part bytes}
```

成功判断：

```text
HTTP 2xx = 当前 part 上传成功
```

失败判断：

```text
HTTP 4xx / 5xx / 网络异常 = 当前 part 失败
```

失败处理：

1. 不要调用 complete。
2. 重新调用 `/part-urls` 获取该 part 的新 URL。
3. 重新 PUT 该 part。
4. 如果客户端进程重启，则重新调用创建或恢复上传接口，服务端会返回已上传 part。

注意：

1. `uploadUrl` 是对象存储地址，不是媒体服务地址。
2. `uploadUrl` 有有效期。
3. 不要把 JSON 发给 `uploadUrl`，body 必须是分片原始二进制。
4. 不要修改分片内容、大小或 partNumber。

## 9. 接口三：重新获取分片上传 URL

当创建接口返回的 `partUrls` 不够，或 URL 过期时调用。

### 9.1 请求

```http
POST {MEDIA_SERVICE_URL}/api/media/files/multipart-uploads/{uploadId}/part-urls
Content-Type: application/json
X-Robot-Id: test001
```

请求体：

```json
{
  "partNumbers": [3, 4, 5, 6]
}
```

### 9.2 响应

```json
{
  "expiresAt": "2026-07-22T15:41:17+08:00",
  "parts": [
    {
      "partNumber": 3,
      "uploadUrl": "http://minio-presigned-url-for-part-3"
    },
    {
      "partNumber": 4,
      "uploadUrl": "http://minio-presigned-url-for-part-4"
    }
  ]
}
```

限制：

```text
一次请求最多数量由 MEDIA_FILE_MAX_PART_URLS_PER_REQUEST 控制，默认 16。
```

建议：

```text
每批获取 16 个 URL，实际 PUT 并发控制在 2 到 4 个。
```

## 10. 接口四：完成分片上传

所有缺失分片都 PUT 成功后调用。

### 10.1 请求

```http
POST {MEDIA_SERVICE_URL}/api/media/files/multipart-uploads/{uploadId}/complete
X-Robot-Id: test001
```

无请求体。

### 10.2 响应

```json
{
  "fileId": "file_abc123",
  "status": "PROCESSING",
  "fileSize": 18268330,
  "ready": false,
  "errorCode": null,
  "errorMessage": null,
  "uploadedAt": "2026-07-22T15:28:00+08:00"
}
```

响应说明：

| 字段 | 说明 |
|---|---|
| `fileId` | 媒体文件 ID |
| `status` | 文件状态 |
| `fileSize` | 文件大小 |
| `ready` | 是否可用 |
| `errorCode` | 错误码 |
| `errorMessage` | 错误信息 |
| `uploadedAt` | 原始文件上传完成时间 |

视频通常返回：

```text
PROCESSING
```

表示原始 MP4 已上传完成，服务端正在进行 HLS 切片等后处理。

## 11. 接口五：查询处理状态

### 11.1 请求

```http
GET {MEDIA_SERVICE_URL}/api/media/files/{fileId}/status
X-Robot-Id: test001
```

### 11.2 响应

```json
{
  "fileId": "file_abc123",
  "status": "READY",
  "fileSize": 18268330,
  "ready": true,
  "errorCode": null,
  "errorMessage": null,
  "uploadedAt": "2026-07-22T15:28:00+08:00"
}
```

状态说明：

| status | 含义 | 客户端动作 |
|---|---|---|
| `UPLOADING` | 原始文件仍在上传 | 通常不应在 complete 后长期停留 |
| `PROCESSING` | 原始文件已完成，服务端处理中 | 继续轮询 |
| `READY` | 文件可用，可播放/下载 | 保存 `fileId`，停止轮询 |
| `FAILED` | 服务端处理失败 | 记录错误，按策略重试或人工处理 |

推荐轮询间隔：

```text
10 到 30 秒
```

## 12. 客户端必须持久化的数据

客户端应在本地保存上传任务清单，至少包含：

```json
{
  "sourceFileId": "test001/task-exec-20260722-001/camera01/visible_20260722_152451.mp4/18268330/1784705200",
  "filePath": "/data/recordings/visible_20260722_152451.mp4",
  "fileSize": 18268330,
  "fileId": "file_abc123",
  "uploadId": "upl_abc123",
  "status": "PROCESSING",
  "error": "",
  "createdAt": "2026-07-22T15:24:51+08:00",
  "updatedAt": "2026-07-22T15:28:00+08:00"
}
```

用途：

1. 进程重启后恢复上传。
2. 避免重复创建上传任务。
3. complete 后继续轮询状态。
4. READY 后可安全清理本地视频。

## 13. 并发建议

建议默认值：

```text
同一机器人同时上传文件数：2
单个文件同时 PUT 分片数：4
单批获取 part URL 数量：16
```

说明：

1. 单批获取 16 个 URL 不等于并发上传 16 个分片。
2. URL 获取批量大小用于减少接口往返。
3. PUT 并发数用于控制带宽、CPU 和对象存储压力。
4. 4G/5G 网络下建议降低 PUT 并发，例如 2。

## 14. 常见错误

### 14.1 400 VALIDATION_ERROR

原因：

```text
fileType、fileName、contentType、fileSize 等必填字段缺失或非法。
```

处理：

```text
检查请求体字段。
```

### 14.2 403 UNTRUSTED_ROBOT_NETWORK

原因：

```text
媒体服务启用了可信机器人网络限制，当前请求来源 IP 不在白名单 CIDR 内。
```

相关配置：

```text
MEDIA_FILE_TRUSTED_ROBOT_NETWORK_ENABLED
MEDIA_FILE_TRUSTED_ROBOT_CIDRS
```

### 14.3 409 INVALID_STATE：准备文件存储桶失败

原因：

```text
媒体服务访问 MinIO/S3 失败，bucketExists 或 makeBucket 失败。
```

检查：

```text
MINIO_ENDPOINT
MINIO_ACCESS_KEY
MINIO_SECRET_KEY
MINIO_BUCKET
MINIO_ENABLED
```

### 14.4 409 SOURCE_FILE_CHANGED

原因：

```text
相同 sourceFileId 对应的 fileName、fileSize 或 fileType 发生变化。
```

处理：

```text
如果是同一个文件，保持字段一致。
如果文件内容已变化，生成新的 sourceFileId。
```

### 14.5 409 UPLOAD_INCOMPLETE

原因：

```text
调用 complete 时，仍有分片没有上传成功。
```

处理：

```text
重新 create/resume，获取 uploadedParts，补传缺失分片后再 complete。
```

### 14.6 409 UPLOAD_SIZE_MISMATCH

原因：

```text
所有分片合计大小与登记的 fileSize 不一致。
```

处理：

```text
检查分片 offset、size、最后一片大小计算。
```

### 14.7 409 UPLOAD_NOT_ACTIVE

原因：

```text
uploadId 已完成、过期或不是 ACTIVE 状态。
```

处理：

```text
重新调用创建或恢复上传接口。
```

### 14.8 429 Too Many Requests

原因：

```text
同一 robotId 或全局 ACTIVE 上传会话数超过限制。
```

默认配置：

```text
MEDIA_FILE_MAX_ACTIVE_UPLOADS_PER_ROBOT=2
MEDIA_FILE_MAX_ACTIVE_UPLOADS_GLOBAL=50
```

常见触发方式：

1. 未传稳定 `sourceFileId`，重复创建多个上传会话。
2. create 后没有继续 PUT 和 complete。
3. 客户端崩溃后留下 ACTIVE 上传，等待过期清理。

处理：

1. 使用稳定 `sourceFileId`。
2. 限制客户端多文件并发。
3. 对 429 做退避重试。
4. 必要时检查数据库 `media_file_upload.status=ACTIVE`。

## 15. 伪代码

```pseudo
file = open(localVideoPath)
fileSize = stat(localVideoPath).size
mtime = stat(localVideoPath).mtime

sourceFileId = robotId + "/" + taskIdOrLocalTaskId + "/" + deviceId + "/" + fileName + "/" + fileSize + "/" + mtime
metadata = json_string({
  "type": "task_recording",
  "cameraId": deviceId,
  "channel": "visible",
  "taskExecutionId": taskExecutionId,
  "localTaskId": localTaskId
})

response = POST /api/media/files/multipart-uploads {
  robotId,
  deviceId,
  sourceFileId,
  fileType: "VIDEO",
  fileName,
  contentType: "video/mp4",
  fileSize,
  metadata
}

save fileId

if response.uploadId is empty:
  while response.status == "PROCESSING":
    sleep
    response = GET /api/media/files/{fileId}/status
  return

uploadedParts = set(response.uploadedParts.partNumber)
missingParts = all part numbers from 1 to partCount excluding uploadedParts

for each batch in missingParts:
  urls = use response.partUrls or POST /multipart-uploads/{uploadId}/part-urls
  for each part in urls with limited concurrency:
    offset = (partNumber - 1) * partSize
    size = min(partSize, fileSize - offset)
    bytes = read local file from offset with size
    PUT uploadUrl bytes

status = POST /api/media/files/multipart-uploads/{uploadId}/complete

while status.status == "PROCESSING":
  sleep
  status = GET /api/media/files/{fileId}/status

if status.status == "READY":
  mark local task READY
else:
  record error
```

## 16. 对接检查清单

开发完成后逐项确认：

1. `MEDIA_SERVICE_URL` 协议和端口正确。
2. `sourceFileId` 稳定且不会随机变化。
3. `metadata` 是 JSON 字符串，不是 JSON 对象。
4. `fileSize` 是真实字节数。
5. partNumber 从 1 开始。
6. 最后一片大小计算正确。
7. PUT body 是分片原始二进制。
8. PUT 失败时重新申请 URL 后重试。
9. 所有 part 成功后才调用 complete。
10. complete 后继续轮询到 READY。
11. 本地持久化 `fileId/uploadId/sourceFileId/status`。
12. 客户端限制并发，避免触发 429。
13. 不上传正在写入的 MP4，建议先写 `.tmp`，完成后 rename。
