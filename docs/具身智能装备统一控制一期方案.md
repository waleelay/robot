# 具身智能装备统一控制一期方案

## 1. 一期目标

一期目标是完成远程控制最小闭环，支撑以下设备组合：

- R1轮式机器人：本体、双光云台、捕网器、警示灯、车灯光、语音对讲。
- G1四足机器狗：本体、双光云台、捕网器、语音对讲。
- G2四足机器狗：本体、双光云台、探照灯。

一期只做平台统一控制骨架和轻量人工接管仲裁，不做复杂机械臂轨迹规划、激光雷达点云流、完整任务调度系统和多厂商插件市场。

## 2. 一期范围

### 2.1 必须实现

| 能力 | 一期要求 |
|---|---|
| 设备注册绑定 | 机器人和上装设备提前注册，前端按绑定设备展示 |
| Effective Profile | 后端返回机器人当前有效控制能力 |
| 控制权 | 支持整机控制权和设备控制权 |
| 控制模式 | 前端实时接收客户端上报的 `MANUAL`、`ASSISTED`、`NAVIGATION` |
| 人工接管 | `NAVIGATION` 或 `ASSISTED` 下，本体手动控制前先走接管接口 |
| 本体遥控 | 前端按 10Hz 发送运动控制帧；点触发少量帧，长按持续发帧 |
| 双光云台 | 前端按 10Hz 发送云台控制帧；点触发少量帧，长按持续发帧 |
| 发射器 | 支持真实安全开关、二次确认、6 发通道发射、审计 |
| 捕网枪 | 前端本地安全开关解锁、二次确认、1 发发射、审计 |
| 多合一喊话设备 | 支持实时喊话、文字转语音、音频文件播放、音量和照明灯控制；媒体流和文件传输不进入普通控制 MQTT payload |
| 急停 | 独立 topic，最高优先级 |
| WebSocket 高频控制 | 本体和云台连续控制走 WebSocket |
| MQTT 控制链路 | 后端按设备类型组装参数并通过 MQTT 下发给 Go 客户端 |
| 客户端状态上报 | Go 客户端通过 `robot/{robotId}/media/client/status` 统一上报在线状态、控制模式、任务状态、摄像头清单和设备状态 |

### 2.2 暂不实现

| 能力 | 放到二期原因 |
|---|---|
| 机械臂复杂控制 | 需要软限位、碰撞、轨迹和姿态策略 |
| 激光雷达点云流和模式控制 | 一期暂不接入雷达控制代码，后续如需接入放到二期 |
| 完整任务调度系统 | 一期只做接管仲裁，不实现任务编排、路线规划和恢复策略 |
| 多厂商 driver 插件化 | 一期可先内置 driver |
| 协议灰度和版本协商 | 一期固定 `embodied-control 1.0` |

## 3. 一期抽象模型

### 3.1 Robot 与 Device 不合并

一期前端展示可以把设备嵌套在机器人下面，但后端领域模型和控制协议不合并。

```text
Robot = 主装备、载体、调度对象、MQTT 路由根节点
Device = Robot 上的可控/可感知能力单元、动作执行目标
```

控制寻址固定为：

```text
robotId + target.deviceId + action
```

示例：

```json
{
  "robotId": "robot-songling-001",
  "target": {
    "scope": "PAYLOAD",
    "deviceId": "ptz-dual-001",
    "deviceType": "DUAL_LIGHT_PTZ"
  },
  "action": "left"
}
```

### 3.2 一期数据来源

一期不在控制服务中新增机器人、设备或能力数据库表，也不返回固定测试设备。控制服务组合管理端登记信息与机器人客户端实时状态：

| 数据 | 一期实现 |
|---|---|
| 机器人及绑定设备 | 从管理端设备接口查询 |
| 平台设备 ID、类型、名称 | 以管理端组件登记信息为准 |
| 动作能力 | 优先使用管理端登记动作；未登记时，仅对已经完成端到端联调的设备类型使用生产兼容动作 |
| 控制 profile | 管理端登记值覆盖生产兼容参数范围 |
| 设备运行状态 | 从 `robot/{robotId}/media/client/status` 的 `devices[].status` 获取，不生成在线、灯光、音量或弹筒等模拟状态 |
| 控制权 session | 控制服务进程内管理 |
| 命令状态 | 控制服务进程内记录并通过 WebSocket 返回发布结果 |

管理端设备查询使用短时缓存，接口暂时不可用时可继续使用最近一次成功结果。客户端状态仍持续上报；每条状态消息经管理端数据合并后只广播一次 `robot.state`。

### 4.5 人工接管

```text
POST /api/control/robots/{robotId}/control-sessions/takeover
```

使用场景：

- 前端实时状态显示 `controlMode=NAVIGATION`，用户要接管本体手动控制。
- 前端实时状态显示 `controlMode=ASSISTED`，用户要切到纯手动控制。
- 前端要开始发送本体 `drive.velocity` 控制帧，但当前模式不是 `MANUAL`。

前端请求：

```json
{
  "fromMode": "NAVIGATION",
  "toMode": "MANUAL",
  "scope": "ROBOT",
  "deviceIds": ["base"],
  "actions": ["drive.velocity"],
  "observedStateSeq": 1288,
  "reason": "manual_takeover"
}
```

请求字段：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `fromMode` | string | 是 | 前端看到的当前模式：`MANUAL` / `ASSISTED` / `NAVIGATION` |
| `toMode` | string | 是 | 目标模式，一期接管固定 `MANUAL` |
| `scope` | string | 是 | 接管范围，本体接管用 `ROBOT` |
| `deviceIds` | array[string] | 是 | 接管设备，本体固定包含 `base` |
| `actions` | array[string] | 是 | 接管后允许的动作，如 `drive.velocity` |
| `observedStateSeq` | number | 是 | 前端最近一次收到的客户端状态序号 |
| `reason` | string | 否 | 接管原因 |

成功响应：

```json
{
  "robotId": "robot-songling-001",
  "controlSessionId": "tc_20260603_0008",
  "previousMode": "NAVIGATION",
  "controlMode": "MANUAL",
  "missionStatus": "PAUSED",
  "scope": "ROBOT",
  "deviceIds": ["base"],
  "actions": ["drive.velocity"],
  "status": "ACTIVE",
  "leaseExpireAt": "2026-06-03T10:31:00+08:00"
}
```

失败响应示例：

```json
{
  "code": "ROBOT_STATE_CHANGED",
  "message": "robot state changed, refresh status before takeover",
  "latestStateSeq": 1290,
  "latestControlMode": "MANUAL"
}
```

后端处理规则：

1. 使用后端缓存的最新客户端状态作为仲裁依据，不能只相信前端传来的 `fromMode`。
2. `observedStateSeq` 小于后端最新 `stateSeq` 时，返回 `ROBOT_STATE_CHANGED`。
3. `NAVIGATION -> MANUAL` 时，后端先下发暂停导航或模式切换 MQTT，再创建控制权。
4. `ASSISTED -> MANUAL` 时，后端下发模式切换 MQTT，再创建控制权。
5. 如果当前已是 `MANUAL` 且无人控制，可直接创建控制权。
6. 如果当前已是 `MANUAL` 且其他终端持有本体控制权，返回 `CONTROL_LOCKED`。
7. 云台控制默认不要求打断 `NAVIGATION`，可走普通设备级控制权。
8. 急停不走接管流程，仍然最高优先级。

#### 4.5.1 控制模式下拉切换

```text
POST /api/control/robots/{robotId}/control-mode
```

使用场景：

- 前端在“装备控制”标题右侧展示控制模式下拉框。
- 下拉框当前值来自机器人端通过 `robot/{robotId}/media/client/status` 实时上报的 `controlMode`。
- 用户手动选择 `MANUAL`、`ASSISTED`、`NAVIGATION` 时调用该接口。
- 机器人端后续再次上报 `controlMode` 后，前端以下发后的实时状态为准重新渲染下拉框。

前端请求：

```json
{
  "controlMode": "NAVIGATION"
}
```

请求字段：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `controlMode` | string | 是 | 目标模式：`MANUAL` / `ASSISTED` / `NAVIGATION` |

后端 MQTT 下发：

```json
{
  "robotId": "robot-songling-001",
  "seq": 1289,
  "target": {
    "deviceId": "base",
    "deviceType": "WHEELED_BASE"
  },
  "action": "control.mode.set",
  "params": {
    "controlMode": "NAVIGATION"
  },
  "issuedAt": "2026-06-03T10:30:30+08:00"
}
```

成功响应：

```json
{
  "status": "PUBLISHED",
  "robotId": "robot-songling-001",
  "controlMode": "NAVIGATION",
  "stateSeq": 1289,
  "issuedAt": "2026-06-03T10:30:30+08:00"
```


## 5. 一期 WebSocket 协议

```text
/ws/control
```

用途：

- 本体摇杆。
- 本体方向键长按。
- 双光云台方向键长按。
- 变焦长按。
- 客户端状态变化推送。

前端发送：

```json

  "type": "control.command",
  "requestId": "req_20260603_0001",
  "payload": {
    "robotId": "robot-songling-001",
    "controlSessionId": "tc_20260603_0001",
    "target": {
      "scope": "BODY",
      "deviceId": "base",
      "deviceType": "WHEELED_BASE"

    "action": "drive.velocity",
```

字段说明：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `robotId` | string | 是 | 机器人 ID |
| `clientId` | string | 否 | 机器人侧客户端 ID |
| `name` | string | 否 | 机器人展示名称 |
| `type` | string | 否 | 机器人类型 |
| `battery` | number | 否 | 电量百分比 |
| `status` | string | 否 | 客户端原始在线状态 |
| `controlMode` | string | 是 | `MANUAL` / `ASSISTED` / `NAVIGATION` |
| `stateSeq` | number | 是 | 客户端状态递增序号，用于接管防旧状态 |
| `currentTaskId` | string | 否 | 当前任务 ID |
| `missionStatus` | string | 否 | `RUNNING` / `PAUSED` / `IDLE` |
| `navigationStatus` | string | 否 | 导航状态 |
| `controlOwner` | object | 否 | 当前控制者；为空表示无人持有 |
| `estopActive` | boolean | 是 | 急停是否生效 |
| `cameras` | array | 否 | 摄像头列表，包含 `cameraId`、`deviceId`、`groupType`、`name`、`quality` |
| `devices` | array | 否 | 本体和上装设备能力列表，结构同控制能力 `devices`；`devices[].status` 用于刷新音量、静音和开关类按钮 |
| `timestamp` | datetime | 是 | 机器人端状态产生时间；缺省时后端补当前时间 |

前端使用规则：

- `controlMode=NAVIGATION` 且要控制本体时，先调用 `4.5 人工接管`。
- `controlMode=ASSISTED` 且要切到纯手动本体控制时，先调用 `4.5 人工接管`。
- `controlMode=MANUAL` 且无人持有控制权时，可直接申请普通控制权。
- `controlMode=NAVIGATION` 时，云台控制默认可申请设备级控制权，不打断导航。
- 前端发起接管时必须携带最近一次收到的 `stateSeq` 作为 `observedStateSeq`。

## 6. 一期 MQTT Topic

| Topic | 方向 | 一期用途 |
|---|---|---|
| `robot/{robotId}/control/body/command` | 后端 -> Go | 机器人本体控制，前后、转向、左移右移 |
| `robot/{robotId}/control/ptz/command` | 后端 -> Go | 双光云台控制，8 方向、自动旋转、变焦等 |
| `robot/{robotId}/control/audio/command` | 后端 -> Go | 扬声器音量控制，滑块调节、音量加减、静音 |
| `robot/{robotId}/control/multi-function/command` | 后端 -> Go | 多合一喊话设备的音量、TTS、文件播放、警报、照明和喊话会话控制 |
| `robot/{robotId}/control/launcher/command` | 后端 -> Go | 发射器控制，6 发类设备 |
| `robot/{robotId}/control/net-gun/command` | 后端 -> Go | 捕网枪控制，1 发类设备 |
| `robot/{robotId}/control/warning-light/command` | 后端 -> Go | 左右警示灯开关控制 |
| `robot/{robotId}/control/vehicle-light/command` | 后端 -> Go | 前后车灯控制，常开、常关、呼吸灯、自定义亮度 |
| `robot/{robotId}/control/safety/estop` | 后端 -> Go | 急停 |
| `robot/{robotId}/media/client/status` | Go -> 后端 | 统一客户端状态上报，包含在线状态、控制模式、任务状态、摄像头清单和设备状态 |

说明：

- 机器人客户端订阅 `robot/{robotId}/control/#`，按 topic 和 payload 中的 `target.deviceId/action` 解析和分发。
- 控制 topic 只保留设备大类，不再携带 `{deviceId}`；具体设备由 payload 中的 `target.deviceId` 标识。
- 前端实时状态由后端消费 `media/client/status` 后通过 WebSocket 推送 `robot.state`。
- 每收到一条 `media/client/status`，控制服务先合并管理端设备标识和能力，再更新机器人注册状态并广播一次 `robot.state`；“一次”指每条上报只产生一条合并事件，不是机器人生命周期只发布一次。
- `controlMode`、`stateSeq`、`cameras` 随 `media/client/status` 上报。
- 后端发布控制命令成功后即可向前端返回 `PUBLISHED`；一期不等待客户端接收确认。

后端 topic 构造规则：

| `deviceType` / action | 下发 topic |
|---|---|
| `WHEELED_BASE`、`QUADRUPED_BASE`、`BIPED_BASE` | `robot/{robotId}/control/body/command` |
| `DUAL_LIGHT_PTZ` | `robot/{robotId}/control/ptz/command` |
| `SPEAKER`、`CLIENT_AUDIO`、`INTERCOM`、`VOLUME_CONTROL` | `robot/{robotId}/control/audio/command` |
| `MULTI_FUNCTION_BROADCASTER` | `robot/{robotId}/control/multi-function/command` |
| `LAUNCHER` | `robot/{robotId}/control/launcher/command` |
| `NET_GUN`、`NET_LAUNCHER` | `robot/{robotId}/control/net-gun/command` |
| `WARNING_LIGHT` | `robot/{robotId}/control/warning-light/command` |
| `VEHICLE_LIGHT`、`SEARCHLIGHT` 或 `light.vehicle.*` | `robot/{robotId}/control/vehicle-light/command` |

`devices[].status` 用于刷新前端有状态控件。一期统一规则：机器人端若提供某个设备的真实状态接口或状态上报，客户端应把结果写入 `devices[].status`，前端必须优先按 `status` 渲染；没有真实状态时只能使用本地临时态或展示“同步中”。控制服务不会生成扬声器音量、发射器连接与弹筒、云台旋转、警示灯或车灯状态。多合一喊话设备的真实定时上报字段为 `volumePercent`、`volumeLimitPercent`、`temperatureC`；文件播放、TTS、警报和照明状态只有在机器人端提供查询/上报接口后才标记为设备真实状态。

设备合并规则：

1. 对外 `devices[].deviceId` 以管理端组件 `code/deviceId/id` 为准。
2. 客户端驱动侧 ID 不一致时，可放在 `devices[].status.driverDeviceId`，用于后端关联运行状态。
3. 后端依次按平台 `deviceId`、`status.driverDeviceId`、唯一 `deviceType` 匹配运行状态。
4. 合并时保留客户端 `status`，以管理端的设备 ID、类型、名称、动作和控制 profile 覆盖平台能力字段。
5. 管理端暂时不可用时使用最近一次成功缓存；没有管理端数据时不伪造设备运行状态。

各设备状态渲染原则：

| 设备 | 状态字段 | 前端渲染规则 |
|---|---|---|
| 双光云台 | `autoRotateEnabled`、`panSpeed`、可扩展 `zoomLevel` | 方向点触/长按不依赖状态；自动旋转按钮必须按 `autoRotateEnabled` 显示开启/停止 |
| 车灯 | `front.mode`、`front.brightness`、`rear.mode`、`rear.brightness` | 如果上报则按前后灯真实模式和亮度渲染；未上报时显示状态同步中并允许控制 |
| 警示灯 | `powerOn` / `enabled`、`mode`、`online` | 如果上报则 switch 按真实开关状态渲染；未上报时显示同步中 |
| 扬声器 | `volumePercent` / `volume`、`muted` | 滑块、静音按钮按状态渲染；按钮加减只计算新的绝对音量 |
| 多合一喊话设备 | `connected`、`volumePercent`、`volumeLimitPercent`、`temperatureC`、`audioFiles` | 音量和温度按设备上报渲染；文件列表按最近一次查询结果渲染；播放、TTS、警报和照明没有真实状态时显示“状态未同步”，不得把前端按钮值伪装成设备真实状态 |
| 捕网枪 | `connected`、可扩展 `ready` / `cooldownRemainingMs` | 若有状态则显示连接/可用状态；若无状态，一期仍使用前端本地安全开关和二次确认 |
| 发射器 | `connected`、`safetySwitchEnabled`、`tubeCount`、`tubes[]` | 必须按真实安全开关和弹筒状态渲染，只有 `LOADED` 且安全开关打开才可发射 |

## 7. 一期 MQTT 命令格式

一期 MQTT 下发给 Go 客户端的是“执行指令”，不是完整业务上下文。接管权、操作者、确认令牌、控制模式、风险策略等字段由后端在下发前完成校验，并记录在后端日志或审计数据中，不作为客户端执行的必需字段。

一期建议使用精简执行 payload。MQTT topic 已经表达了机器人和设备域，payload 中只保留客户端执行需要的字段。

```json
{
  "robotId": "robot-songling-001",
  "seq": 1024,
  "target": {
    "deviceId": "base",
    "deviceType": "WHEELED_BASE"
  },
  "action": "drive.velocity",
  "params": {
    "linearX": 0.3,
    "linearY": 0,
    "angularZ": -0.2
  },
  "issuedAt": "2026-06-03T10:30:00+08:00"
}
```

字段说明：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `robotId` | string | 是 | 控制哪台机器人，也与 MQTT topic 中的 `{robotId}` 一致 |
| `seq` | number | 建议 | 前端递增序号，客户端可用于日志和观察丢帧/乱序 |
| `target.deviceId` | string | 是 | 控制哪个设备，例如 `base`、`ptz-dual-001`、`net-gun-001` |
| `target.deviceType` | string | 是 | 设备类型，客户端按类型分发处理 |
| `action` | string | 是 | 平台动作名，例如 `drive.velocity`、`left`、`fire` |
| `params` | object | 是 | 后端校验、限幅后的动作参数 |
| `issuedAt` | datetime | 是 | 后端下发时间 |

以下字段不进入一期 MQTT 执行指令：

| 字段 | 不下发原因 |
|---|---|
| `protocol` | topic 已按控制域隔离，一期客户端只消费控制 topic，可不再用 payload 字段识别协议 |
| `version` | 一期协议固定，暂不做版本协商；后续多版本兼容时再恢复或放到 topic/header |
| `messageType` | 当前 topic 只承载命令，不混传事件或 ack |
| `commandId` | 无客户端 ack 和去重要求时可不下发；后端可在日志、接口响应中内部保留 |
| `operator` | 操作者信息用于后端鉴权、审计，不参与客户端执行 |
| `controlSessionId` | 控制权由后端校验，客户端不再二次判断 |
| `controlMode` | 控制模式由后端结合客户端状态判断，客户端只执行已放行的指令 |
| `traceId` | 链路追踪保留在后端日志即可，一期客户端不依赖 |
| `target.vendor` / `target.model` | 设备归属由客户端配置或后端 profile 管理，执行指令只要求设备 ID 和类型 |
| `policy` | 风险等级、二次确认、独占控制、过期策略由后端处理 |
| `params.frameRateHz` | 10Hz 是前端发送频率，不是动作参数；速度由 `params` 中的速度值决定 |

## 8. 一期 Action 参数

### 8.1 连续控制类

连续控制类不区分点触、长按或结束状态。前端统一按 10Hz 发送控制帧；点触就是发送少量帧，长按就是持续发送帧。后端不解释点触或长按语义，只校验、限幅并按设备类型组装 MQTT 参数。

#### `drive.velocity`

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `linearX` | number | 是 | 前后速度，正数向前 |
| `linearY` | number | 否 | 横移速度；不支持横移的设备由后端裁剪为 0 |
| `angularZ` | number | 是 | 转向角速度 |

#### 双光云台方向动作：`up`、`down`、`left`、`right`、`left_up`、`right_up`、`left_down`、`right_down`

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `speed` | number | 是 | 方向移动速度，后端按云台能力限幅；建议 `10~40` |
| `duration` | number | 是 | 单帧移动持续时间，单位秒；建议 `0.2~0.8` |

#### `camera.zoom`

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `zoomSpeed` | number | 是 | 变焦速度，正数放大，负数缩小 |

#### `ptz.auto_rotate`

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `enabled` | boolean | 是 | true 开启云台自动旋转，false 停止自动旋转 |
| `panSpeed` | number | 否 | 自动旋转水平速度，默认 0.3，由后端按云台能力限幅 |

### 8.2 离散动作类

#### 发射器 `fire`

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `tube` | number | 是 | 发射器弹筒编号，范围 `1~6` |
| `waitStatusAfterFire` | boolean | 否 | 发射后是否等待/查询最新状态，默认 `true` |
| `keepSafetyOn` | boolean | 否 | 发射后是否保持 safety 状态，默认 `false` |
| `confirmToken` | string | 是 | 二次确认 token |

#### 发射器 `set_safety`

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `safety_on` | boolean | 是 | true 打开安全开关，false 关闭 |
| `wait_status` | boolean | 否 | 设置后是否等待/查询最新状态，默认 `true` |

#### 发射器 `get_status`

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `temporarilyEnableSafety` | boolean | 否 | 查询前是否临时打开安全开关，默认 `true` |
| `restoreSafetyAfterQuery` | boolean | 否 | 查询后是否恢复原安全开关状态，默认 `true` |

#### 捕网器 `fire`

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `confirmToken` | string | 是 | 二次确认 token |

一期约束：发射器 `LAUNCHER` 的真实安全开关通过 `set_safety` 下发；捕网枪 `NET_GUN` 页面上的安全开关只作为前端本地解锁按钮，不调用后端接口，不下发 MQTT。捕网器可能没有状态查询接口，一期不要求捕网器状态查询。

#### 扬声器 `set_volume`

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `volumePercent` | number | 是 | 绝对音量百分比，范围 0-100；前端滑块、加号、减号都换算成该绝对值下发 |

#### 扬声器 `set_mute`

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `mute` | boolean | 是 | true 静音，false 取消静音 |

#### 多合一喊话设备

设备类型固定为 `MULTI_FUNCTION_BROADCASTER`。它是一个物理 `Device`，通过 `actions` 暴露多种能力，不拆成喊话器、TTS、播放器和照明灯四个虚拟设备。

| action | params | 必填 | 说明 |
|---|---|---:|---|
| `set_volume` | `volumePercent: number` | 是 | 设置绝对音量，范围 `0~100`；后端还应按 `controlProfile.maxVolumePercent` 限幅 |
| `start_broadcast` | `mediaSessionId: string` | 是 | 开始实时喊话会话；仅传媒体会话标识，不传 PCM/Opus 二进制数据 |
| `stop_broadcast` | `{}` | 是 | 停止实时喊话会话 |
| `play_tts` | `text: string`、`voice: string`、`loop: boolean` | 是 | 播放文字；`voice=MALE/FEMALE`，`loop` 表示是否循环 |
| `stop_tts` | `{}` | 是 | 停止循环 TTS；底层当前只明确支持停止循环播放 |
| `list_audio_files` | `{}` | 是 | 查询设备已有音频文件列表，查询结果随统一状态上报刷新 `status.audioFiles` |
| `play_audio_file` | `fileName: string`、`loop: boolean` | 是 | 播放设备本地音频文件，文件名使用 UTF-8 |
| `stop_audio_file` | `{}` | 是 | 停止当前音频文件播放 |
| `delete_audio_file` | `fileName: string` | 是 | 删除设备本地音频文件；后端应校验文件名来自已查询列表 |
| `play_alarm` | `{}` | 是 | 播放设备内置警报 |
| `stop_alarm` | `{}` | 是 | 停止设备内置警报 |
| `light.set` | `enabled: boolean`、`brightness?: number`、`mode?: string` | 是 | 控制多合一设备照明灯；当前设备文档未给底层映射，机器人端接口确认前不得假定编码 |
| `set_tilt` | `value: number` | 是 | 设置设备俯仰值，设备文档范围 `80~220`；仅在管理端注册该能力时展示 |

音频文件上传不定义为 MQTT action。文件二进制应走独立 HTTP 文件传输接口，成功上传到机器人客户端后，由客户端调用设备 `POST /upload-file`；普通控制命令中只传文件名等元数据。

#### `light.set`

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `enabled` | boolean | 是 | true 打开，false 关闭 |
| `brightness` | number | 否 | 亮度 0-100 |
| `mode` | string | 否 | `STEADY` 常亮，`FLASH` 闪烁 |

#### `light.vehicle.set`

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `front` | object | 是 | 前灯完整状态 |
| `front.mode` | string | 是 | `OFF` 常关、`ON` 常开、`BREATH` 呼吸灯、`CUSTOM` 自定义亮度 |
| `front.brightness` | number | 是 | 前灯亮度，范围 0-100；仅 `mode=CUSTOM` 时有效，其余模式传 0 |
| `rear` | object | 是 | 后灯完整状态 |
| `rear.mode` | string | 是 | `OFF` 常关、`ON` 常开、`BREATH` 呼吸灯、`CUSTOM` 自定义亮度 |
| `rear.brightness` | number | 是 | 后灯亮度，范围 0-100；仅 `mode=CUSTOM` 时有效，其余模式传 0 |

## 9. 前端到 MQTT 参数构造

本章说明前端如何构造请求参数，后端如何根据设备类型解析并组装 MQTT 下发参数。客户端收到后如何抽象 ROS2 调用不在本文档范围内，由客户端同事负责。

示例使用 JSONC 写注释，实际 HTTP、WebSocket、MQTT 传输时必须去掉注释，使用标准 JSON。

### 9.1 通用构造规则

前端只传“当前控制帧”或“一次性动作意图”，不传厂商底层协议字段。

连续控制类：

```text
前端点触：按 10Hz 发送少量控制帧。
前端长按：按 10Hz 持续发送控制帧。
前端结束操作：不再发送控制帧。
后端：每收到一帧就校验、限幅、组装 MQTT 并下发。
客户端：消费后端参数并调用 ROS2 或设备 driver。
```

后端负责补齐：

| 后端补齐字段 | 来源 | 说明 |
|---|---|---|
| `seq` | 前端 `client.seq` | 终端递增序号 |
| `issuedAt` | 后端时间 | 后端下发时间 |

说明：`protocol`、`version`、`messageType`、`commandId` 不进入一期 MQTT 执行 payload；其中 `commandId` 如需排查问题，可由后端保留在接口响应和日志中。

后端内部处理但不下发 MQTT：

| 内部字段 | 用途 |
|---|---|
| `controlSessionId` | 校验控制权、接管关系 |
| `controlMode` | 校验当前模式下是否允许该动作 |
| `operator` | 鉴权、审计、日志 |
| `confirmToken` | 捕网器等高风险动作的二次确认校验 |
| `policy` | 后端内部风险策略、确认策略、冷却策略 |
| `traceId` | 后端链路日志追踪 |

前端必须传：

| 前端字段 | 说明 |
|---|---|
| `robotId` | 控制哪台机器人 |
| `controlSessionId` | 已申请到的控制权 ID；不需要控制权的动作可为空 |
| `target.scope` | 控制目标范围，如 `BODY`、`PAYLOAD`、`SENSOR` |
| `target.deviceId` | 控制哪一个设备 |
| `target.deviceType` | 设备类型，用于后端选择参数构造策略 |
| `controlMode` | 控制模式：`MANUAL` / `ASSISTED` / `NAVIGATION`；本体手动控制通常为 `MANUAL` |
| `action` | 平台动作名 |
| `params` | 动作参数 |
| `client.terminalId` | 当前终端 ID |
| `client.source` | 控件来源 |
| `client.seq` | 当前终端递增序号 |
| `client.timestamp` | 前端发送时间 |

后端处理流程：

```text
1. 解析 HTTP/WebSocket 请求体
2. 解析 CurrentUser：userId、orgId、roles、clientId
3. 从管理端登记信息或其最近一次成功缓存中校验 robotId
4. 校验 target.deviceId 是否属于该机器人已登记组件
5. 根据 target.deviceType 选择参数构造策略
6. 校验 action 是否在 devices[].actions 中
7. 校验 controlMode 与后端缓存的最新客户端状态是否允许执行
8. 校验 controlSessionId 是否有效，或判断该 action 是否无需控制权
9. 按 action 校验 params
10. 按 controlProfile 裁剪速度、角速度、亮度等参数
11. 对高风险 action 校验 confirmToken、安全开关、冷却时间等后端策略
12. 生成 commandId、issuedAt
13. 组装简化 MQTT 执行 payload
14. 按设备类型发布到对应设备域 topic，急停发布到 safety/estop topic
15. 通过 WebSocket 给前端回 accepted/rejected
```

后端设备类型构造策略：

| `deviceType` | 典型 action | 后端构造重点 |
|---|---|---|
| `WHEELED_BASE` | `drive.velocity` | 不支持横移时 `linearY=0`，按轮式底盘限速 |
| `QUADRUPED_BASE` | `drive.velocity` | 允许横移时保留 `linearY`，按四足限速 |
| `DUAL_LIGHT_PTZ` | `up`、`down`、`left`、`right`、`left_up`、`right_up`、`left_down`、`right_down`、`ptz.auto_rotate`、`camera.zoom` | 裁剪方向移动速度和持续时间、自动旋转速度、变焦速度 |
| `LAUNCHER` | `get_status`、`set_safety`、`fire` | 安全开关真实下发，发射按 1 到 6 弹筒构造 |
| `NET_GUN` | `fire` | 前端本地安全开关解锁后才能点击；后端校验 confirmToken |
| `SEARCHLIGHT` | `light.set` | 裁剪亮度，校验模式 |
| `WARNING_LIGHT` | `get_state`、`set_state`、`set_mode` | 按 `lightId` 构造单灯或全部灯查询、开关和模式参数 |
| `VEHICLE_LIGHT` | `light.vehicle.set` | 前后灯作为一个整体设备，后端只下发平台通用 `mode/brightness` 语义 |
| `MULTI_FUNCTION_BROADCASTER` | `set_volume`、`start_broadcast`、`stop_broadcast`、`play_tts`、`stop_tts`、`list_audio_files`、`play_audio_file`、`stop_audio_file`、`delete_audio_file`、`play_alarm`、`stop_alarm`、`light.set`、`set_tilt` | 只保留平台语义参数；TCP/HTTP 前缀、十六进制音量、Opus 分帧和设备地址由机器人客户端适配 |

后端组装伪代码：

```text
intent = parse(frontendPayload)
profile = managementProfile.find(robotId, intent.target.deviceId)
builder = commandBuilderFactory.get(intent.target.deviceType)
params = builder.validateAndBuildParams(intent.action, intent.params, profile.controlProfile)
checkControlSession(intent.controlSessionId, intent.target, intent.action)
checkControlMode(intent.controlMode, latestRobotState, intent.action)
checkInternalPolicy(intent.action, intent.params, currentUser)

mqttPayload = {
  robotId: intent.robotId,
  seq: intent.client.seq,
  target: {
    deviceId: intent.target.deviceId,
    deviceType: intent.target.deviceType
  },
  action: intent.action,
  params: params,
  issuedAt: now()
}
```

### 9.4 发射器与捕网枪：`get_status`、`set_safety`、`fire`

使用场景：

- 发射器打开/关闭真实安全开关，并通过接口下发给客户端。
- 发射器二次确认后按 1 到 6 弹筒发射。
- 捕网枪页面保留本地安全开关按钮，只用于解锁前端发射按钮，不调用后端接口。
- 捕网枪二次确认后触发发射；捕网器可能没有状态接口，一期不强制查询状态。

发射器和捕网枪动作建议走 REST `/commands`，不走高频 WebSocket。

发射器渲染规则：

- `controlProfile.tubes` 表示平台登记的弹筒能力范围，例如 `[1,2,3,4,5,6]`，只用于兜底生成按钮。
- `status.connected`、`status.safetySwitchEnabled`、`status.tubeCount`、`status.tubes[]` 表示客户端上报的真实运行态，前端渲染与禁用逻辑必须优先使用 `status`。
- 当前没有 `status.tubes[]` 时，前端可以按 `controlProfile.tubes` 展示按钮，但状态应显示“未知”，且不能判定为可发。
- 只有 `safetySwitchEnabled=true` 且目标弹筒 `state=1 / LOADED` 时，发射按钮才可点击。

客户端通过统一状态 topic `robot/{robotId}/media/client/status` 回写发射器状态，示例：

```jsonc
{
  "deviceId": "launcher_38mm",
  "deviceType": "LAUNCHER",
  "status": {
    "connected": true,                           // 发射器是否连接
    "safetySwitchEnabled": false,                // 真实安全开关状态
    "tubeCount": 6,                              // 弹筒数量
    "tubes": [
      {
        "tube": 1,                               // 弹筒编号
        "state": 1,                              // 0 空，1 已装填，2 发射中，3 堵塞，255 未知
        "stateName": "LOADED"                    // EMPTY / LOADED / FIRING / BLOCKED / UNKNOWN
      }
    ]
  },
  "controlProfile": {
    "tubes": [1, 2, 3, 4, 5, 6],                 // 能力配置，不等于实时弹筒状态
    "requiresConfirm": true,                     // 发射前需要二次确认
    "requiresSafetySwitch": true                 // 发射前需要打开真实安全开关
  }
}
```

### 9.9 语音对讲：沿用媒体对讲链路

一期语音对讲建议沿用当前实时视频模块的对讲设计，不纳入 `embodied-control` 普通设备控制 MQTT。

原因：

- 对讲使用 LiveKit 音频链路，不是普通设备动作。
- 当前项目已有视频会话绑定对讲的接口和 MQTT topic。
- 前端点击对讲按钮时，应走媒体模块的 intercom start/stop，而不是 `/commands`。

一期调用方式：

```text
POST /api/control/robots/{robotId}/cameras/{deviceId}/video/intercom/start
POST /api/control/video-sessions/{sessionId}/intercom/stop
```

如果二期需要把语音对讲也纳入统一设备目录，可增加：

```text
target.scope = AUDIO
target.deviceId = intercom-001
action = audio.intercom.start / audio.intercom.stop
```

但一期不建议迁移，避免和现有 LiveKit 对讲链路重复。

### 9.10 多合一喊话设备

#### 9.10.1 建模和能力注册

多合一设备虽然包含喊话、文字转语音、音频播放和照明灯等能力，但在平台注册时仍是一个物理设备：

```text
deviceType = MULTI_FUNCTION_BROADCASTER
```

前端不得按设备名称硬编码面板。前端查询 `control-profile` 后，根据该设备的 `actions` 和 `controlProfile` 动态显示控件。能力示例：

```jsonc
{
  "deviceId": "broadcaster-001",                 // 平台内该物理上装的唯一 ID
  "bindingId": "bind-r1-broadcaster-001",        // 与当前机器人的绑定关系
  "scope": "PAYLOAD",                            // 机器人上装
  "deviceType": "MULTI_FUNCTION_BROADCASTER",    // 多合一喊话设备
  "displayName": "多合一喊话照明设备",             // 前端显示名称
  "onlineStatus": "online",                      // 客户端上报的设备在线状态
  "enabled": true,                               // 平台是否允许使用
  "actions": [
    "set_volume",                                // 设置音量
    "start_broadcast",                           // 开始实时喊话
    "stop_broadcast",                            // 停止实时喊话
    "play_tts",                                  // 播放文字转语音
    "stop_tts",                                  // 停止循环 TTS
    "list_audio_files",                          // 查询设备音频文件
    "play_audio_file",                           // 播放设备音频文件
    "stop_audio_file",                           // 停止文件播放
    "delete_audio_file",                         // 删除设备音频文件
    "play_alarm",                                // 播放警报
    "stop_alarm",                                // 停止警报
    "light.set",                                 // 控制照明灯；底层接口确认后启用
    "set_tilt"                                   // 控制设备俯仰；管理端注册后才展示
  ],
  "controlProfile": {
    "minVolumePercent": 0,                       // 平台允许的最小音量
    "maxVolumePercent": 100,                     // 平台允许的最大音量，实际还受设备 volumeLimitPercent 限制
    "voices": ["MALE", "FEMALE"],               // TTS 可选音色
    "audioFileFormats": ["wav", "mp3", "aac", "m4a", "flac"],
    "liveBroadcast": {
      "codec": "OPUS",                          // 机器人客户端输出给设备的编码
      "sampleRateHz": 8000,                      // 设备协议要求 8kHz
      "channels": 1,                             // 单声道
      "frameDurationMs": 60                      // 每帧 60ms，480 个样本
    },
    "tilt": {
      "min": 80,                                 // 设备协议允许的最小俯仰值
      "max": 220                                 // 设备协议允许的最大俯仰值
    }
  }
}
```

管理端登记的 action 优先决定前端展示与后端校验。为保证已与机器人端完成联调的 MQTT 数据结构不发生变化，本体、云台、扬声器、发射器、捕网器和警示灯在管理端尚未登记 action 时可使用生产兼容动作；该兼容表只补动作和参数范围，不补设备运行状态。多合一设备等未纳入兼容表的类型仍要求管理端登记 action；照明灯底层接口尚未定义时，正式能力数据中不应启用 `light.set`，或将控件标记为不可用。

#### 9.10.2 通用控制结构

多合一设备的低频动作统一调用：

```text
POST /api/control/robots/{robotId}/commands
```

以 TTS 播放为例，前端请求：

```jsonc
{
  "controlSessionId": "tc_20260721_0010",        // 当前设备级控制权
  "target": {
    "scope": "PAYLOAD",                          // 上装设备
    "deviceId": "broadcaster-001",               // 平台设备 ID，不是底层 IP
    "deviceType": "MULTI_FUNCTION_BROADCASTER"   // 后端据此选择构造器和 MQTT topic
  },
  "action": "play_tts",                          // 具体动作：播放 TTS
  "params": {
    "text": "请立即离开危险区域",                  // 待播放文本，UTF-8，不能为空
    "voice": "FEMALE",                           // MALE 男声，FEMALE 女声
    "loop": true                                 // true 循环播放，false 单次播放
  },
  "client": {
    "terminalId": "web-client-001",              // 当前终端 ID
    "source": "broadcaster_tts_button",          // 前端控件来源
    "seq": 10001,                                // 当前终端递增序号
    "timestamp": "2026-07-21T14:30:00+08:00"     // 前端产生动作的时间
  }
}
```

后端完成设备归属、action、控制权、参数和长度校验后，下发：

```jsonc
{
  "robotId": "robot-songling-001",               // 目标机器人
  "seq": 10001,                                  // 沿用前端序号，便于日志定位
  "target": {
    "deviceId": "broadcaster-001",               // 目标物理设备
    "deviceType": "MULTI_FUNCTION_BROADCASTER"   // 客户端选择设备适配器
  },
  "action": "play_tts",                          // 客户端按 action 分发
  "params": {
    "text": "请立即离开危险区域",                  // 已校验文本
    "voice": "FEMALE",                           // 平台音色枚举
    "loop": true                                 // 是否循环
  },
  "issuedAt": "2026-07-21T14:30:00+08:00"        // 后端发布时间
}
```

MQTT topic：

```text
robot/robot-songling-001/control/multi-function/command
```

设备 IP、TCP/HTTP 端口、协议前缀和厂商编码均不得由前端传入，也不进入 MQTT payload。它们属于机器人客户端的设备适配配置，客户端通过 `target.deviceId` 找到对应配置。

#### 9.10.3 各动作 params

| action | 前端和 MQTT 的 `params` | 校验和处理 |
|---|---|---|
| `set_volume` | `{"volumePercent": 60}` | 必须为整数 `0~100`；后端按 profile 和最新 `volumeLimitPercent` 限幅 |
| `start_broadcast` | `{"mediaSessionId": "ms_001"}` | 媒体会话必须属于当前 robot/device/terminal；普通 MQTT 不携带音频帧 |
| `stop_broadcast` | `{}` | 停止当前设备的实时喊话转发 |
| `play_tts` | `{"text": "你好", "voice": "MALE", "loop": false}` | 文本非空；音色必须在 profile 中；文本长度上限由能力配置给出 |
| `stop_tts` | `{}` | 当前底层协议只明确停止循环 TTS |
| `list_audio_files` | `{}` | 客户端查询后通过统一状态上报刷新 `status.audioFiles` |
| `play_audio_file` | `{"fileName": "notice.mp3", "loop": true}` | 文件名必须来自最近查询结果；禁止目录路径和路径穿越字符 |
| `stop_audio_file` | `{}` | 停止当前文件播放 |
| `delete_audio_file` | `{"fileName": "notice.mp3"}` | 文件名必须来自最近查询结果；删除前端应二次确认 |
| `play_alarm` | `{}` | 播放设备内置警报 |
| `stop_alarm` | `{}` | 停止设备内置警报 |
| `light.set` | `{"enabled": true, "brightness": 80, "mode": "STEADY"}` | 上层保持通用灯光语义；机器人端接口确认后由客户端完成映射 |
| `set_tilt` | `{"value": 120}` | 整数范围 `80~220`，客户端映射为底层无符号字节 |

音量、TTS 和照明等按钮均属于离散低频动作，不按 10Hz 重复发送。只有实时喊话的音频媒体帧按媒体编码节奏持续传输。

#### 9.10.4 实时喊话的数据面

实时喊话必须拆分为控制面和媒体数据面：

```text
前端创建/取得媒体会话
  -> 前端下发 start_broadcast(mediaSessionId)
  -> 后端校验并发布控制 MQTT
  -> 机器人客户端订阅对应媒体会话
  -> 客户端将音频转换为 8000Hz、16bit、单声道 PCM
  -> 客户端按 480 样本/60ms 编码为 Opus
  -> 客户端按设备协议发送 [42] + Opus 二进制帧
  -> 前端停止时下发 stop_broadcast
  -> 客户端停止向设备发送音频帧
```

禁止把以下内容放入普通 MQTT JSON：

- PCM 或 Opus 的 Base64 数据。
- `[42]` 等设备协议前缀。
- 采集缓冲区、帧序号等媒体内部字段。
- 设备 IP 和端口。

这样可以避免 JSON/Base64 膨胀、MQTT 控制消息阻塞，也能让未来其他厂商设备复用同一 `start_broadcast/stop_broadcast` 语义。

#### 9.10.5 音频文件上传

设备文档中的 `/upload-file` 是机器人局域网内的设备接口，平台后端通常不能直接访问。上层应将文件上传定义为独立数据传输流程，不复用 `/commands`，建议接口为：

```text
POST /api/control/robots/{robotId}/devices/{deviceId}/audio-files
Content-Type: multipart/form-data
```

流程为：平台校验文件扩展名、大小和设备归属，将文件传给对应机器人客户端；客户端再调用设备侧 `POST /upload-file`。上传结果和最新文件列表通过任务结果或统一状态上报返回。具体文件分片、断点续传和失败重试可在机器人端上传接口确定后补充。

#### 9.10.6 客户端设备协议映射

客户端适配层按以下规则解析平台命令。表中的底层数据只用于客户端实现，不进入前端或平台 MQTT：

| 平台 action / 状态 | 设备侧协议映射 |
|---|---|
| `set_volume` | TCP 发送 `[14]${volHex}`；`volumePercent` 转为两位大写十六进制，例如 `9 -> 09`、`100 -> 64` |
| 音量状态 | 每 2 秒解析 `[99]` 后的 JSON；`volume_real -> volumePercent`、`volume_limit -> volumeLimitPercent`、`temperature -> temperatureC` |
| `list_audio_files` | HTTP `GET /fetch-files`；成功后把文件名数组写入 `devices[].status.audioFiles` |
| `play_audio_file` | TCP 发送 `[12]${loopFlag}${fileNameUtf8}`；`loop=true -> 1`，否则 `0` |
| `stop_audio_file` | TCP 发送 `[13]` |
| `delete_audio_file` | HTTP `POST /del-file`，表单字段 `filename=params.fileName` |
| `play_tts` | 非循环使用 `[31]${voiceCode}${text}`，循环使用 `[32]${voiceCode}${text}`；`MALE -> 0`，`FEMALE -> 1` |
| `stop_tts` | TCP 发送 `[17]`；当前设备文档只明确停止循环 TTS |
| `play_alarm` | TCP 发送 `[18]` |
| `stop_alarm` | TCP 发送 `[19]` |
| `start_broadcast` | 建立媒体转发；每 60ms 向设备发送 `[42]` 加一帧 Opus 二进制数据 |
| `stop_broadcast` | 停止发送实时喊话音频帧，底层无额外停止报文 |
| `set_tilt` | TCP 端口 `12345` 发送两个字节：`0x8D` 和 `params.value` |
| `light.set` | 待机器人端补充照明灯开关、亮度、模式以及状态查询接口后定义；客户端不得自行猜测编码 |

原设备协议还提供“收音”能力：`[40]` 开始、`[41]` 停止，返回 16000Hz、单声道、20ms Opus 帧。该能力不属于本次前端四项功能，一期暂不注册对应 action；后续需要监听时应复用媒体数据面设计，不把音频二进制放入控制 MQTT。

#### 9.10.7 状态上报和前端渲染

机器人客户端通过现有统一 topic 上报状态：

```text
robot/{robotId}/media/client/status
```

多合一设备状态示例：

```jsonc
{
  "deviceId": "broadcaster-001",                 // 平台设备 ID
  "deviceType": "MULTI_FUNCTION_BROADCASTER",    // 设备类型
  "onlineStatus": "online",                      // 设备连接状态
  "status": {
    "connected": true,                           // TCP/HTTP 连接是否可用
    "volumePercent": 58,                         // volume_real，设备真实生效音量
    "volumeLimitPercent": 100,                   // volume_limit，设备允许的最大音量
    "temperatureC": 23.56,                       // temperature，摄氏温度
    "audioFiles": ["notice.mp3", "alarm.wav"],   // 最近一次成功查询的设备文件列表
    "audioFilesUpdatedAt": "2026-07-21T14:30:02+08:00",
    "observedAt": "2026-07-21T14:30:02+08:00"    // 状态采集时间
  }
}
```

前端渲染规则：

- 音量滑块最大值取 `min(controlProfile.maxVolumePercent, status.volumeLimitPercent)`，当前值取 `status.volumePercent`。
- 温度和在线状态直接按设备上报显示。
- 音频文件下拉列表取 `status.audioFiles`；列表未查询成功时显示“文件列表未同步”。
- 设备协议没有播放、TTS、警报和照明查询接口时，前端不得仅因命令返回 `PUBLISHED` 就显示“设备已生效”。可以显示“指令已下发”，但真实状态应保持未知。
- 后续机器人端增加真实状态接口后，再在 `status.playback`、`status.tts`、`status.alarm`、`status.light` 下扩展，不修改现有命令 envelope。

#### 9.10.8 待机器人端确认项

在开始真实设备联调前，机器人端仍需明确：

1. 照明灯的开关、亮度、模式和状态查询接口。
2. TTS 单次播放是否支持中途停止；当前 `[17]` 只明确停止循环播放。
3. 文件上传大小限制、同名覆盖规则、中文文件名和特殊字符规则。
4. 实时喊话的连接复用、断线重连、发送缓冲和设备忙时处理。
5. 文件播放、TTS、警报是否互斥，以及新动作到达时的抢占优先级。
6. 是否提供当前播放类型、文件名、循环状态和照明状态的真实查询或主动上报。

## 10. 控制模式与接管策略

### 10.1 控制模式

| 模式 | 说明 | 一期处理 |
|---|---|---|
| `MANUAL` | 手动模式，用户直接控制本体或设备 | 本体控制帧必须处于该模式 |
| `ASSISTED` | 辅助模式，机器人或算法参与辅助控制 | 本体手动控制前需要接管到 `MANUAL` |
| `NAVIGATION` | 导航模式，机器人执行导航或任务 | 本体手动控制前需要接管到 `MANUAL`；云台可默认不打断导航 |

### 10.2 接管策略

| 最新客户端状态 | 前端动作 | 后端策略 |
|---|---|---|
| `MANUAL` 且无人控制 | 申请本体控制 | 允许创建控制权 |
| `MANUAL` 且他人控制本体 | 申请本体控制 | 拒绝，返回 `CONTROL_LOCKED` |
| `ASSISTED` | 手动控制本体 | 先调用 takeover，切到 `MANUAL` 后返回控制权 |
| `NAVIGATION` | 手动控制本体 | 先调用 takeover，暂停导航后返回控制权 |
| `NAVIGATION` | 控制云台 | 默认允许申请设备级控制权，不打断导航 |
| `NAVIGATION` | 捕网器发射 | 默认禁止，除非任务策略明确允许 |
| 任意模式 | 急停 | 允许，最高优先级 |
| 急停生效中 | 普通控制 | 禁止 |

### 10.3 接管时序

```text
机器人客户端上报 media/client/status
  -> 控制服务合并管理端设备 ID、动作与控制 profile
  -> 后端缓存最新 stateSeq/controlMode
  -> WebSocket 广播一次 robot.state
  -> 前端展示“接管”按钮
  -> 用户点击接管
  -> 前端调用 /control-sessions/takeover，携带 observedStateSeq
  -> 后端比对 observedStateSeq 与最新 stateSeq
  -> 后端按策略暂停导航或切换模式
  -> 后端创建 controlSessionId
  -> 前端开始按 10Hz 发送 drive.velocity 控制帧
```

注意：前端实时状态只用于展示和发起接管，最终是否允许接管由后端根据最新客户端状态、用户权限、控制权和策略决定。

## 11. 一期频率和安全

| 控制对象 | 前端发送频率 | 后端处理 |
|---|---:|---|
| 本体点触 | 10Hz，发送少量帧 | 每帧校验、限幅、按本体设备类型组装 MQTT |
| 本体长按/摇杆 | 10Hz，按住期间持续发送 | 每帧校验、限幅、按本体设备类型组装 MQTT |
| 云台点触 | 10Hz，发送少量帧 | 每帧校验、限幅、按云台设备类型组装 MQTT |
| 云台长按 | 10Hz，按住期间持续发送 | 每帧校验、限幅、按云台设备类型组装 MQTT |
| 变焦长按 | 10Hz，按住期间持续发送 | 每帧校验、限幅、按云台/相机设备类型组装 MQTT |

安全规则：

- 连续控制类不要求前端发送结束帧。
- 后端不区分点触、长按、松手，只处理收到的每一帧控制数据。
- 本体手动控制必须处于 `MANUAL`，`NAVIGATION` 或 `ASSISTED` 下需先接管。
- 发射器 `fire` 必须有控制权、confirmToken、真实安全开关和冷却校验。
- 捕网枪 `fire` 必须有控制权和 confirmToken；安全开关只在前端本地解锁按钮，不作为后端校验项。
- 急停使用 `robot/{robotId}/control/safety/estop`，不进入普通命令队列。
- 客户端如何消费控制帧并调用 ROS2，由客户端同事单独抽象实现。

## 12. 一期验收标准

1. 前端能根据 `control-profile` 动态展示不同机器人绑定的上装控制面板。
2. 两个终端不能同时控制同一机器人本体。
3. 一个终端控制本体时，另一个终端可按配置控制云台。
4. 本体和云台点触时，前端能按 10Hz 发送少量控制帧，后端逐帧组装 MQTT。
5. 本体和云台长按时，前端能按 10Hz 持续发送控制帧，后端逐帧组装 MQTT。
6. 发射器没有 confirmToken 或真实安全开关时不能发射；捕网枪没有 confirmToken 或前端本地安全开关未打开时不能点击发射。
7. `NAVIGATION` 或 `ASSISTED` 下手动控制本体时，前端必须先接管成功再发送控制帧。
8. 前端接管请求携带旧 `observedStateSeq` 时，后端返回 `ROBOT_STATE_CHANGED`。
9. 后端能记录命令发布，并通过 `media/client/status` 刷新机器人在线状态、控制模式、任务状态、摄像头清单和设备状态。
10. 多合一喊话设备能按 `actions` 动态展示能力；音量、TTS、文件播放、警报和俯仰命令使用统一 payload，下层 TCP/HTTP 编码不暴露给前端；实时音频和上传文件不进入普通控制 MQTT。
11. 多合一喊话设备的音量、音量上限和温度按真实上报渲染；照明等缺少状态接口的能力只能显示“指令已下发”或“状态未同步”。
