## 1. 文档说明

### 1.1 编制目的

本文档用于统一描述系统现有业务能力、业务流程、功能范围和验证标准，作为功能确认、系统联调和交付验收的依据。

### 1.2 文档范围

本文档第 4 章覆盖以下已具备代码实现的能力：

1. 机器人实时视频；
2. 实时视频语音对讲；
3. 机器人主动呼叫中心端；
4. 通用文件上传、存储与播放；
5. 具身智能装备统一控制一期；
6. 大屏 BFF 已有聚合接口；
7. 大屏统计接口与报告任务。

第 5 章单独记录已经形成方案、但尚未形成完整实现的固定摄像头任务视频、多合一喊话设备控制和统一控制扩展能力。

### 1.3 状态说明

第 4 章描述已有代码承载的功能，第 5 章描述已经形成方案但尚未形成完整实现的能力，两类内容不作为同一完成状态。涉及真实机器人、音视频设备、对象存储、转码组件及外部业务数据的功能，以对应环境的联调结果作为最终可用性依据。

### 1.4 系统协作边界

本文档以媒体服务、统一控制服务和大屏 BFF 的服务端能力为主要范围。以下模块由协作方负责开发，文档中涉及这些模块的内容用于说明接口调用、数据交互和联调依赖，不作为本服务端范围内的代码交付内容。

| 协作模块 | 主要职责 | 与本系统的交互 |
| --- | --- | --- |
| 设备端客户端（终端协作模块） | 接入机器人摄像头、麦克风、扬声器及上装设备，执行媒体和控制指令 | 接收 MQTT 指令，发布音视频，上报设备状态、呼叫状态和执行结果 |
| 控制前端 | 提供实时视频、语音对讲和设备控制操作界面 | 调用媒体与控制接口，连接媒体房间，接收 WebSocket 状态事件 |
| 管理端后台服务（配置数据提供方） | 维护机器人、设备、能力、绑定关系及相关业务配置 | 向媒体服务、控制服务和大屏 BFF 提供设备档案、能力配置和业务关联数据 |

后续章节中的“设备端客户端”“控制前端”和“管理端后台服务”均指上述协作模块。设备端客户端属于双向通信的终端接入方；管理端后台服务属于配置数据提供方。

## 2. 系统概述

### 2.1 系统背景

平台由媒体服务、控制服务、大屏 BFF 以及协作方负责的设备端客户端、控制前端和管理端后台服务等模块组成，用于承载机器人实时视频、语音对讲、文件媒体、设备控制、大屏态势数据及统计报告功能。

### 2.2 已有能力概览

| 能力域 | 已有能力 |
| --- | --- |
| 机器人实时视频 | 服务端创建视频会话、生成播放凭证、发送摄像头与清晰度切换指令、关闭会话 |
| 实时视频对讲 | 服务端管理视频会话中的对讲状态并发送对讲启停指令 |
| 机器人主动呼叫 | 服务端接收设备端呼叫，处理中心端接听、拒绝、超时和结束 |
| 通用文件服务 | 简单上传、分片上传、续传、文件查询、视频处理与播放信息查询 |
| 统一控制一期 | 控制画像、设备能力、控制会话、接管、释放和设备指令下发 |
| 大屏 BFF | 总览、设备分组、设备详情、任务、告警、处置及实时事件适配 |
| 大屏统计与报告 | 统计查询接口、报告任务创建、状态查询和报告文件下载 |

## 3. 总体业务流程

### 3.1 实时视频流程

1. 用户从设备列表或业务页面选择机器人和摄像头；
2. 前端向控制服务提交视频启动请求；
3. 控制服务调用媒体服务创建视频会话；
4. 媒体服务向设备端客户端发送视频启动指令；
5. 设备端客户端发布对应摄像头媒体流；
6. 前端获取播放凭证并订阅媒体流；
7. 用户关闭视频后，服务端通知设备端客户端停止发布并结束会话。

### 3.2 视频对讲流程

1. 用户在已建立的视频会话中开启对讲；
2. 系统创建或更新对讲状态，并通知设备端客户端开启音频；
3. 浏览器发布本地麦克风音频并订阅机器人音频；
4. 用户可以执行本地静音或关闭对讲；
5. 对讲关闭后，视频会话可继续保持。

### 3.3 机器人主动呼叫流程

1. 协作方设备端客户端通过消息通道发起呼叫；
2. 控制服务创建呼叫记录并通过 WebSocket 推送中心端来电；
3. 中心端可以接听或拒绝来电；
4. 接听成功后，系统调用媒体服务建立对讲会话；
5. 呼叫结束时，系统关闭关联对讲并推送呼叫终态；
6. 在规定时间内未接听的呼叫自动转为超时状态。

### 3.4 文件上传与播放流程

1. 调用方创建直接上传或分片上传任务；
2. 服务端保存文件或分片，并记录上传状态；
3. 分片完整后完成合并；
4. 视频文件按配置进入媒体信息解析或转码处理；
5. 调用方查询文件与处理状态；
6. 前端使用返回的文件或 HLS 播放信息进行播放。

### 3.5 设备控制流程

1. 前端查询机器人控制画像及设备能力；
2. 用户申请控制会话，或对已有控制状态执行接管；
3. 前端根据设备能力组织动作及参数；
4. 控制服务校验机器人、设备、动作和参数；
5. 控制服务通过 MQTT 发布设备执行指令；
6. 控制服务通过 WebSocket 发布控制指令事件和机器人状态；
7. 用户完成操作后释放控制会话。

### 3.6 大屏数据流程

1. 大屏前端向 BFF 查询总览、设备、任务和告警信息；
2. BFF 调用任务系统、告警系统、管理端后台服务及其他外部业务数据接口，并转换为统一返回结构；
3. 没有来源的数据字段返回空值或空集合；
4. 设备、任务和告警变化通过实时事件通道推送；
5. 前端根据对象标识更新地图、列表和详情。

## 4. 功能需求

### 4.1 机器人实时视频

#### REQ-VIDEO-001 创建实时视频会话

- **需求描述**：用户选择机器人、摄像头和视频清晰度后，系统创建实时视频会话。
- **现有规则**：
  1. 视频会话记录机器人、摄像头、清晰度、创建人、组织和客户端信息；
  2. 会话创建后向设备端客户端发送视频启动指令；
  3. 会话状态随客户端响应和媒体轨道状态更新；
  4. 创建失败时返回失败状态及原因。
- **验证标准**：会话能够完成创建、客户端启动、媒体连接和状态更新过程。

#### REQ-VIDEO-002 获取播放凭证

- **需求描述**：前端通过视频会话获取浏览器连接媒体房间所需的播放凭证。
- **现有规则**：普通查看者获得仅订阅媒体的凭证；具备媒体操作角色的用户获得支持麦克风发布的交互凭证。
- **验证标准**：前端能够使用凭证进入对应媒体房间并订阅机器人媒体轨道。

#### REQ-VIDEO-003 切换摄像头与清晰度

- **需求描述**：视频会话中支持切换机器人摄像头及主、子码流。
- **现有规则**：切换请求更新会话目标，并向设备端客户端发送新的启动或切换指令。
- **验证标准**：切换完成后，会话中的摄像头和清晰度信息与请求一致。

#### REQ-VIDEO-004 关闭视频会话

- **需求描述**：用户可以主动关闭视频会话。
- **现有规则**：关闭时更新会话状态，向设备端客户端发送停止指令，并停止关联的客户端录制任务。
- **验证标准**：关闭后会话进入结束状态，服务端完成停止指令发布。

### 4.2 实时视频语音对讲

#### REQ-ICM-001 开启对讲

- **需求描述**：用户可以在视频会话中开启机器人与浏览器之间的语音对讲。
- **现有规则**：系统向设备端客户端发送对讲启动指令，并更新视频会话中的对讲状态。
- **验证标准**：对讲状态能够从启动中更新为已连接或失败，双方音频轨道可按状态建立。

#### REQ-ICM-002 静音与关闭对讲

- **需求描述**：对讲过程中支持浏览器本地静音和关闭对讲。
- **现有规则**：关闭对讲时发送停止指令并更新对讲状态；关闭对讲不直接结束视频会话。
- **验证标准**：静音状态与页面操作一致；关闭后对讲进入结束状态，视频仍可继续播放。

### 4.3 机器人主动呼叫中心端

#### REQ-CALL-001 接收与推送来电

- **需求描述**：服务端接收设备端客户端发起的中心端呼叫，并向控制前端推送来电事件。
- **现有规则**：
  1. 服务端创建唯一呼叫标识；
  2. 同一机器人存在有效呼叫或对讲时，不重复创建新呼叫；
  3. 来电通过 WebSocket 推送到中心端；
  4. 呼叫记录包含机器人、呼叫状态、创建时间和过期时间。
- **验证标准**：中心端收到与机器人请求一致的来电事件，重复呼叫按现有忙线规则处理。

#### REQ-CALL-002 接听、拒绝与超时

- **需求描述**：中心端可以接听或拒绝呼叫，未处理呼叫在到期后自动超时。
- **现有规则**：
  1. 接听操作校验媒体操作角色；
  2. 只有处于响铃状态且未过期的呼叫可以接听；
  3. 接听成功后调用媒体服务建立对讲；
  4. 呼叫的状态变化同步推送到设备端客户端和控制前端。
- **验证标准**：接听、拒绝和超时均形成对应呼叫状态；已处理来电不能再次接听。

#### REQ-CALL-003 结束呼叫

- **需求描述**：已接听的呼叫可以结束。
- **现有规则**：结束呼叫时关闭关联对讲，更新呼叫状态并推送结果。
- **验证标准**：结束后呼叫和对讲均进入终态，服务端完成面向设备端客户端和控制前端的状态发布。

### 4.4 通用文件上传、存储与播放

#### REQ-FILE-001 直接上传

- **需求描述**：系统提供普通文件直接上传接口。
- **现有规则**：上传成功后生成文件记录，保存文件标识、名称、大小、内容类型、存储位置和业务关联信息。
- **验证标准**：文件内容成功保存，调用方能够通过文件标识查询上传结果。

#### REQ-FILE-002 分片上传与续传

- **需求描述**：大文件支持初始化上传、上传分片、查询已完成分片、完成合并和取消上传。
- **现有规则**：
  1. 初始化后返回唯一上传标识；
  2. 服务端记录已完成的分片；
  3. 调用方可根据已完成分片继续上传；
  4. 完成请求触发文件合并并生成文件记录。
- **验证标准**：中断后能够查询并继续上传，所有分片完成后生成完整文件。

#### REQ-FILE-003 文件查询与播放

- **需求描述**：系统提供文件信息、媒体处理状态和播放信息查询。
- **现有规则**：视频文件可按配置生成 HLS 播放资源；调用方通过文件状态判断原文件或 HLS 资源是否可用。
- **验证标准**：已完成处理的视频能够取得播放信息，处理失败时能够查询失败状态。

### 4.5 具身智能装备统一控制一期

#### REQ-CTRL-001 查询控制画像

- **需求描述**：前端可以查询机器人的类型、厂商、型号、在线状态、控制模式、状态序号及设备能力列表。
- **现有规则**：设备能力列表包含设备标识、设备类型、支持动作及控制参数配置。
- **验证标准**：接口返回与系统配置一致的机器人和设备控制信息。

#### REQ-CTRL-002 申请控制会话

- **需求描述**：用户可以按机器人和设备范围申请控制会话。
- **现有规则**：
  1. 控制会话记录机器人、用户、客户端、设备范围、动作范围和租约到期时间；
  2. 与其他客户端的有效会话发生设备范围冲突时返回占用信息；
  3. 同一客户端重复申请冲突范围时续期现有会话。
- **验证标准**：无冲突时返回有效会话；存在冲突时返回当前占用信息。

#### REQ-CTRL-003 接管与释放控制会话

- **需求描述**：前端可以接管机器人控制状态，并在完成操作后释放控制会话。
- **现有规则**：
  1. 接管请求可携带已观察的机器人状态序号；
  2. 状态序号已变化时返回刷新提示；
  3. 接管后机器人控制模式更新为人工控制，任务状态更新为暂停；
  4. 释放操作将控制会话状态更新为已释放。
- **验证标准**：接管和释放后返回的会话状态与服务端内存状态一致。

#### REQ-CTRL-004 下发设备控制指令

- **需求描述**：前端可以向机器人已有设备下发其能力列表中声明的动作。
- **现有规则**：
  1. 服务端校验机器人、设备和动作是否存在；
  2. 移动、云台、变焦、灯光、扬声器、发射器和捕网器等已有动作按对应规则构造参数；
  3. 数值型参数按设备控制画像进行范围限制；
  4. 指令通过 MQTT 发布到设备端客户端；
  5. 发布后生成指令标识并通过 WebSocket 推送已发布事件。
- **验证标准**：支持的动作生成对应 MQTT 载荷；不支持的动作被拒绝；返回结果包含指令标识和发布状态。

多合一喊话设备的完整控制能力尚未形成端到端实现，其设计范围见第 5.2 节。

### 4.6 大屏 BFF 已有接口

#### REQ-BFF-001 全景总览与设备查询

- **需求描述**：BFF 提供全景总览、设备分组和设备详情查询接口。
- **现有规则**：
  1. BFF 从管理端后台服务、任务系统和告警系统等外部业务数据接口读取设备及相关业务数据；
  2. 返回结构按大屏模型进行字段转换；
  3. 没有真实来源的字段返回空值、空集合或未知状态；
  4. 不使用固定模拟值补齐缺失业务数据。
- **验证标准**：接口能够返回统一结构；缺失字段保持空值语义。

#### REQ-BFF-002 任务与告警查询

- **需求描述**：BFF 提供任务列表、告警列表、告警详情及告警处置适配接口。
- **现有规则**：任务数据由任务系统接口提供，告警数据与处置能力由告警系统接口提供，BFF 完成字段映射和返回结构转换。
- **验证标准**：接口响应结构符合大屏前端约定，处置请求能够传递到告警系统处置接口。

#### REQ-BFF-003 实时事件适配

- **需求描述**：BFF 接收并转发设备、任务和告警相关实时事件。
- **现有规则**：实时事件通过 WebSocket 适配层提供给大屏前端。
- **验证标准**：任务系统、告警系统或设备状态数据源的事件到达后，大屏连接能够收到对应事件载荷。

### 4.7 大屏统计与报告

#### REQ-STAT-001 统计接口

- **需求描述**：系统提供总览指标、趋势和排行等统计查询接口。
- **现有规则**：接口支持接收统计筛选条件，并以约定结构返回已有统计结果；没有统计数据时返回空值或空集合。
- **验证标准**：各统计接口能够正常响应，空数据不会被固定数值替代。

#### REQ-STAT-002 报告任务

- **需求描述**：系统支持创建报告任务、查询报告状态和下载已生成报告。
- **现有规则**：
  1. 创建报告任务后返回报告标识；
  2. 报告任务记录生成状态；
  3. 报告成功后可以通过下载接口获取本地生成的 PDF 文件；
  4. 报告失败时保留失败状态和错误信息。
- **验证标准**：报告任务能够完成创建、状态查询和成功文件下载流程。

### 5.2 多合一喊话设备控制

#### 5.2.1 需求目标

将喊话、文字转语音、音频文件播放、警报、音量、照明和俯仰等能力作为一个物理上装设备统一接入平台。前端根据设备注册的能力动态展示控制项，平台通过统一控制入口下发低频动作，实时音频和文件内容分别复用媒体服务与文件服务。

#### 5.2.2 设备建模

多合一喊话设备在平台中注册为一个物理设备：

```text
deviceType = MULTI_FUNCTION_BROADCASTER
```

设备能力由 `actions` 和 `controlProfile` 描述，不按照设备名称在前端硬编码。设计动作包括：

| 能力分类 | 设计动作 | 说明 |
| --- | --- | --- |
| 音量控制 | `set_volume` | 设置设备播放音量 |
| 实时喊话 | `start_broadcast`、`stop_broadcast` | 建立或停止实时喊话会话 |
| 文字转语音 | `play_tts`、`stop_tts` | 播放或停止文字合成语音 |
| 音频文件 | `list_audio_files`、`play_audio_file`、`stop_audio_file`、`delete_audio_file` | 查询、播放、停止和删除设备音频文件 |
| 警报播放 | `play_alarm`、`stop_alarm` | 播放或停止设备内置警报 |
| 照明控制 | `light.set` | 控制多合一设备照明灯 |
| 俯仰控制 | `set_tilt` | 调整设备俯仰角度 |

#### 5.2.3 设计流程

1. 协作方管理端后台服务登记设备类型、设备标识、机器人绑定关系、支持动作和参数范围；
2. 协作方控制前端查询机器人控制画像，根据设备 `actions` 动态展示可用控制项；
3. 音量、TTS、文件播放、警报、照明和俯仰等低频动作通过统一设备命令接口提交；
4. 控制服务根据设备类型和动作构造设备控制消息；
5. 设备端客户端中的多合一设备驱动将统一消息转换为设备 TCP、HTTP 或厂商协议；
6. 设备端客户端上报设备在线状态、音量、音量上限、温度及能够获取的播放状态；
7. 协作方控制前端使用设备上报状态刷新控件，不将本地按钮状态作为设备真实状态。

#### 5.2.4 媒体与文件边界

1. 实时喊话音频不放入普通控制消息，由媒体链路传输，设备端客户端负责解码并转换为设备要求的音频格式；
2. TTS 文本可通过控制动作提交，生成的音频由媒体服务或设备端适配流程处理；
3. 音频文件内容通过文件服务上传和保存，控制消息只传递文件标识或可访问地址；
4. 普通控制消息只承载动作、参数和业务标识，不直接承载大块音频或文件二进制内容。

#### 5.2.5 已有基础

1. 协作方控制前端已有多合一设备页面组件；
2. 媒体服务已有独立 TTS 音频服务；
3. 平台已有通用文件上传、存储与播放能力；
4. 统一控制一期已有控制画像、设备动作和 MQTT 指令骨架。

上述基础尚未按 `MULTI_FUNCTION_BROADCASTER` 设备模型贯通为完整链路。

#### 5.2.6 尚未形成的闭环

1. 控制画像中尚未注册完整的多合一设备类型、动作和参数范围；
2. 控制服务尚未实现多合一设备各动作的专用参数构造与路由；
3. 协作方设备端客户端尚未形成对应设备驱动及厂商协议转换；
4. 实时喊话尚未完成从浏览器、媒体服务到实际喊话设备的完整链路；
5. TTS、文件播放、警报、照明和俯仰尚未完成真实设备执行与状态回传；
6. 前端页面尚未与真实设备能力和状态形成完整绑定。

#### 5.2.7 实现完成条件

1. 设备能够通过控制画像返回真实注册能力；
2. 前端只展示设备已注册的动作；
3. 每类控制动作均能从控制前端到达设备端客户端中的设备驱动；
4. 设备驱动能够调用真实设备协议并返回明确执行结果；
5. 音量、温度和可查询的播放状态能够按设备真实数据展示；
6. 实时喊话、TTS 和音频文件播放能够在真实设备上完成端到端验证；
7. 照明或俯仰等底层协议未明确的能力不进入可用动作列表。

### 5.3 固定监控摄像头任务实时视频

#### 5.3.1 需求目标

任务可以关联一个或多个固定监控摄像头，用户能够在任务详情中查看与作业区域对应的实时视频，补充机器人自身摄像头之外的现场画面。

#### 5.3.2 设计范围

1. 建立任务、作业区域与固定摄像头的关联关系；
2. 通过摄像头网关统一管理摄像头连接、鉴权、拉流和协议转换；
3. 前端通过平台视频会话获取播放信息，不直接访问摄像头原始账号和地址；
4. 支持任务内摄像头列表、视频启动、摄像头切换、停止和异常状态展示；
5. 复用现有媒体播放能力承载浏览器侧播放。

#### 5.3.3 尚未形成的闭环

1. 尚未建立正式摄像头网关服务；
2. 尚未建立任务与固定摄像头的权威映射；
3. 尚未形成固定摄像头视频会话创建接口；
4. 尚未形成网关拉流、协议转换和前端播放的完整代码链路。

#### 5.3.4 实现完成条件

1. 任务能够查询到准确的关联摄像头；
2. 在线摄像头能够通过平台会话在浏览器播放；
3. 摄像头离线、连接失败和资源不足时返回明确状态；
4. 关闭播放后摄像头网关和媒体资源能够释放。

### 5.4 统一控制扩展能力

#### 5.4.1 设计范围

统一控制扩展方案在一期控制骨架基础上覆盖以下能力：

1. 机械臂关节、末端姿态、夹爪和轨迹控制；
2. 激光雷达点云、扫描状态和相关设备控制；
3. 更完整的任务暂停、人工接管和恢复流程；
4. 多厂商设备驱动插件化及协议版本管理；
5. 控制命令、设备状态、任务和媒体的关联记录与回放；
6. 面向复杂设备的参数约束、安全策略和状态模型。

#### 5.4.2 与一期边界

一期已具备统一控制入口、设备能力、控制会话、MQTT 指令和状态事件等基础骨架。本节所列复杂设备模型、驱动插件、轨迹类控制及关联回放能力尚未纳入一期代码范围。

#### 5.4.3 实现完成条件

1. 新设备类型能够通过统一能力模型注册；
2. 控制服务能够按设备类型将指令路由到对应驱动；
3. 协作方设备端客户端驱动能够适配至少一种真实机械臂或激光雷达设备；
4. 复杂指令具备参数校验、状态反馈和异常结果；
5. 任务接管及恢复能够与真实任务状态保持一致；
6. 命令、状态、任务和媒体能够按统一标识关联查询。
