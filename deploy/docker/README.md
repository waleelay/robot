# Docker deployment

This directory provides one-command package, install, and uninstall entrypoints for:

- Media Service (`backend`)
- Control Service (`control-service`)
- Bigscreen BFF (`bigscreen-bff`)
- LiveKit Server (`livekit/livekit-server`)
- LiveKit Egress (`livekit/egress`)

The default configuration deploys only the three application services. External middleware such as MySQL, Redis, EMQX, LiveKit, MinIO, and Elasticsearch is expected to be reachable from the container through the values in `.env`.

## First run

```bash
cd deploy/docker
cp .env.example .env
```

Edit `.env` for the target host, especially database passwords and middleware addresses.

Application packages are extracted to:

```text
/opt/qihang/robot/dockerWorkspace
```

The path can be changed with `APP_WORKSPACE_ROOT` in `.env`. Each container mounts its service directory to `/app`:

```text
/opt/qihang/robot/dockerWorkspace/media-service    -> /app
/opt/qihang/robot/dockerWorkspace/control-service  -> /app
/opt/qihang/robot/dockerWorkspace/bigscreen-bff    -> /app
/opt/qihang/robot/dockerWorkspace/livekit-server   -> /etc/livekit
/opt/qihang/robot/dockerWorkspace/livekit-egress   -> /out
```

This keeps `bin`, `boot`, `config`, `lib`, and `logs` outside the container, so later partial updates can be done by replacing files under the host workspace and restarting the corresponding container.

LiveKit config files are generated during install:

```text
/opt/qihang/robot/dockerWorkspace/livekit-server/livekit.yaml
/opt/qihang/robot/dockerWorkspace/livekit-egress/egress.yaml
```

By default, repeated installs do not overwrite existing runtime directories or existing LiveKit config files:

```text
INSTALL_MODE=skip_existing
```

For a full package upgrade that overwrites the extracted Java runtime files and regenerated LiveKit config, set:

```text
INSTALL_MODE=overwrite
```

LiveKit exposes:

```text
Signaling/API: 7880/tcp
RTC TCP:       7881/tcp
RTC UDP:       50000-50100/udp by default
```

For LAN or production use, make sure the host firewall opens the configured UDP range and that `LIVEKIT_URL` is reachable by browser and robot clients.

## Script roles

```text
package.sh            Build Java packages, build app images, and create the installer tar.gz.
install.sh            Load image archives, extract packages, generate config, and start services.
uninstall.sh          Stop services.
load-images.sh        Internal helper used by install.sh.
prepare-workspace.sh  Internal helper used by install.sh.
```

`docker-compose.build.yml` is only used by `package.sh` on the build machine. The installer uses `docker-compose.yml`, which has no `build` section, so the server does not need source code or Maven.

## Installer Package

Put pre-downloaded third-party Docker image archives such as LiveKit images under:

```text
deploy/docker/tool-images
```

Java service tar packages are generated by `package.sh`; you do not need to place them manually. In the final installer they are stored under:

```text
packages/
```

Docker image archives, including the copied LiveKit image archives, are stored under:

```text
images/
```

Supported file suffixes are `.tar`, `.tar.gz`, `.tgz`, `.tar.xz`, `.txz`, `.tar.zst`, and `.tzst`. For example:

```bash
docker pull livekit/livekit-server:latest
docker pull livekit/egress:latest
mkdir -p deploy/docker/tool-images
docker save livekit/livekit-server:latest | gzip -9 > deploy/docker/tool-images/livekit-server.tar.gz
docker save livekit/egress:latest | gzip -9 > deploy/docker/tool-images/livekit-egress.tar.gz
```

`livekit/egress` contains large browser and media-processing dependencies. If gzip does not reduce the file much, use `xz` for a smaller transfer package:

```bash
docker save livekit/egress:latest | xz -T0 -9e > deploy/docker/tool-images/livekit-egress.tar.xz
```

Or use `zstd` when compression and decompression speed matter more:

```bash
docker save livekit/egress:latest | zstd -T0 --ultra -22 -o deploy/docker/tool-images/livekit-egress.tar.zst
```

Create the installer on the build machine:

```bash
cd deploy/docker
./package.sh
```

The package is written to:

```text
dist/robot-mediaserver-installer-<timestamp>.tar.gz
```

Upload it to the server, then:

```bash
tar -xzf robot-mediaserver-installer-<timestamp>.tar.gz
cd robot-mediaserver-installer-<timestamp>
cp .env.example .env
./install.sh
```

The installer does not compile Java code on the server and does not require source directories. It loads Docker image archives from `images/`, extracts Java service packages from `packages/`, generates LiveKit config files, and starts Compose. With the default `INSTALL_MODE=skip_existing`, existing runtime directories and existing LiveKit config files are left untouched.

## One-command install or upgrade

```bash
./install.sh
```

This starts the services in the background from the packaged images and Java runtime packages.

## One-command uninstall

```bash
./uninstall.sh
```

To also remove Compose-managed volumes:

```bash
./uninstall.sh --volumes
```

## Service URLs

```text
Media Service:     http://localhost:8088
Control Service:   http://localhost:8082
Bigscreen BFF:     http://localhost:8090
```

Inside the Compose network, services should call each other by service name:

```text
http://media-service:8088
http://control-service:8082
ws://control-service:8082/ws/control
```
