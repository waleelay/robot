# Python 客户端任务视频上传对接文档

## 1. 对接目标

机器人执行巡检/巡逻任务后，会在本地生成一个或多个任务视频文件。Python 客户端需要把这些视频上传到媒体服务，媒体服务负责：

1. 创建或恢复分片上传任务。
2. 返回 MinIO/S3 预签名分片上传地址。
3. 接收客户端完成通知并合并原始视频。
4. 对视频做后处理，例如 HLS 切片、封面、预览图。
5. 返回 `fileId`，供后续任务产物查询、播放和下载使用。

本文只说明“任务视频上传”对接，不赘述任务执行流程、任务结束事件和前端播放流程。

## 2. 当前接口边界

媒体服务接口地址：

```text
{MEDIA_SERVICE_URL}/api/media/files
```

Python 客户端默认配置：

```text
MEDIA_SERVICE_URL=http://175.155.35.79:8088
RECORDING_UPLOAD_ENABLED=true
RECORDING_DIRECTORY=./recordings
RECORDING_MANIFEST_PATH=./recording-upload-manifest.json
RECORDING_DEVICE_ID=camera01
RECORDING_UPLOAD_FILE_CONCURRENCY=2
RECORDING_UPLOAD_PART_CONCURRENCY=4
RECORDING_UPLOAD_PART_URL_BATCH_SIZE=16
```

上传使用 multipart 接口。普通任务视频不要走单接口上传，因为任务视频可能较大，需要断点续传和并发分片。

## 3. 客户端整体流程

```text
任务视频文件生成完成
  -> Python 客户端扫描 RECORDING_DIRECTORY
  -> 为本地文件生成稳定 sourceFileId
  -> 写入本地 manifest，状态 PENDING
  -> POST /multipart-uploads 创建或恢复上传
  -> 根据 uploadedParts 跳过已上传分片
  -> 获取 part 上传 URL
  -> 并发 PUT 分片到 uploadUrl
  -> POST /complete 完成原始文件上传
  -> 视频进入 PROCESSING
  -> GET /{fileId}/status 轮询
  -> READY 后后端可播放，客户端可按本地缓存策略清理文件
```

`complete` 成功只代表原始 MP4 已经传完并合并，不代表马上能播放。视频通常还要等待服务端 HLS 切片完成。

## 4. 本地文件发现规则

当前 Python 客户端上传模块位于：

```text
python-client/robot_media_client/recordingupload/
```

核心文件：

| 文件 | 职责 |
|---|---|
| `runner.py` | 扫描本地文件、创建任务、驱动上传状态机 |
| `client.py` | 封装媒体服务上传接口 |
| `uploader.py` | 并发 PUT 缺失分片 |
| `manifest.py` | 本地断点续传清单 |

`Runner.discover()` 当前只扫描 `RECORDING_DIRECTORY` 第一层普通文件：

```text
跳过目录
跳过隐藏文件
跳过 0 字节文件
不递归子目录
```

任务视频建议在录像文件完全写入完成后，再移动到 `RECORDING_DIRECTORY`，避免正在写入的 MP4 被提前发现。更稳妥的做法是：

```text
先写入 visible_20260722_152451.mp4.tmp
写完并关闭文件
rename 为 visible_20260722_152451.mp4
```

## 5. sourceFileId 规则

`sourceFileId` 是断点续传和幂等恢复的关键，必须传。

不能只传：

```json
{
  "robotId": "test111",
  "fileName": "visible_20260722_152451.mp4"
}
```

否则服务端无法判断这是不是同一个本地源文件，重复调用创建接口时容易产生多个 ACTIVE 上传会话，并触发 429。

推荐格式：

```text
{robotId}/{taskExecutionId或manual}/{deviceId}/{fileName}/{fileSize}/{mtime}
```

示例：

```text
test111/task-exec-20260722-001/camera01/visible_20260722_152451.mp4/18268330/1784705200
```

如果任务执行记录 ID 还没有生成，可以先使用本地任务临时 ID 或任务批次 ID：

```text
test111/local-task-20260722-1524/camera01/visible_20260722_152451.mp4/18268330/1784705200
```

注意：

1. 同一本地文件重试上传时，必须使用相同 `sourceFileId`。
2. 同名文件被覆盖后，如果内容变化，建议让 `fileSize` 或 `mtime` 变化，从而生成新的 `sourceFileId`。
3. 不要每次重试都随机生成 `sourceFileId`。

当前 Python 客户端默认生成规则在 `runner.py`：

```text
{recordingDeviceId}/{fileName}/{fileSize}/{mtime}
```

如果要完整表达任务维度，建议同事扩展为包含 `robotId`、任务执行 ID 或本地任务批次 ID。

## 6. metadata 建议

`metadata` 是字符串字段，建议传 JSON 字符串。任务视频建议至少包含：

```json
{
  "type": "task_recording",
  "cameraId": "camera01",
  "channel": "visible",
  "taskExecutionId": "task-exec-20260722-001",
  "localTaskId": "local-task-20260722-1524",
  "recordingStartedAt": "2026-07-22T15:24:51+08:00",
  "recordingEndedAt": "2026-07-22T15:27:12+08:00"
}
```

字段说明：

| 字段 | 是否建议 | 说明 |
|---|---|---|
| `type` | 是 | 固定传 `task_recording`，便于和手动录像、抓拍区分 |
| `cameraId` | 是 | 摄像头设备 ID |
| `channel` | 是 | 如 `visible`、`thermal` |
| `taskExecutionId` | 有则传 | 任务执行记录 ID |
| `localTaskId` | 建议 | 任务执行 ID 未生成时的本地临时关联 ID |
| `recordingStartedAt` | 建议 | 视频开始时间 |
| `recordingEndedAt` | 建议 | 视频结束时间 |

当前 Python `recordingupload.Client.create_or_resume()` 只传了基础字段。如果任务视频需要传 `metadata`、`robotId`、`extensionId`，需要扩展该方法入参和请求体。

## 7. 第一步：创建或恢复上传

请求：

```http
POST {MEDIA_SERVICE_URL}/api/media/files/multipart-uploads
Content-Type: application/json
X-Robot-Id: test111
```

请求体示例：

```json
{
  "robotId": "test111",
  "deviceId": "camera01",
  "extensionId": null,
  "sourceFileId": "test111/task-exec-20260722-001/camera01/visible_20260722_152451.mp4/18268330/1784705200",
  "fileType": "VIDEO",
  "fileName": "visible_20260722_152451.mp4",
  "contentType": "video/mp4",
  "fileSize": 18268330,
  "metadata": "{\"type\":\"task_recording\",\"cameraId\":\"camera01\",\"channel\":\"visible\",\"taskExecutionId\":\"task-exec-20260722-001\"}"
}
```

字段说明：

| 字段 | 是否必填 | 说明 |
|---|---|---|
| `robotId` | 建议必填 | 机器人 ID；也可通过 `X-Robot-Id` 传，body 优先 |
| `deviceId` | 建议必填 | 摄像头设备 ID |
| `extensionId` | 否 | 通用扩展 ID，没有可传 `null` |
| `sourceFileId` | 强烈建议必填 | 本地源文件幂等键 |
| `fileType` | 是 | 视频固定 `VIDEO` |
| `fileName` | 是 | 原始文件名 |
| `contentType` | 是 | MP4 传 `video/mp4` |
| `fileSize` | 是 | 字节数，必须大于 0 |
| `metadata` | 建议 | JSON 字符串 |

响应示例：

```json
{
  "fileId": "file_abc123",
  "uploadId": "upl_abc123",
  "uploadMode": "MULTIPART",
  "status": "UPLOADING",
  "partSize": 16777216,
  "partCount": 2,
  "uploadedParts": [],
  "partUrls": [
    {
      "partNumber": 1,
      "uploadUrl": "http://minio-presigned-url-for-part-1"
    },
    {
      "partNumber": 2,
      "uploadUrl": "http://minio-presigned-url-for-part-2"
    }
  ],
  "expiresAt": "2026-07-25T15:26:17+08:00"
}
```

响应处理：

| 情况 | 客户端动作 |
|---|---|
| `uploadId` 非空 | 继续上传缺失分片 |
| `uploadId` 为空且 `status=PROCESSING/READY` | 原文件已上传过，不再 PUT 分片，只保存 `fileId` 并轮询状态 |
| `uploadedParts` 非空 | 跳过这些 part，只上传缺失 part |
| `partUrls` 非空 | 可以直接使用首批 URL，也可以继续调用 `/part-urls` 获取 |

当前 Python 封装没有解析响应里的 `partUrls`，而是统一再调用 `/part-urls`。这不是错误，只是多一次接口；如果要优化，可以把 `partUrls` 加入 `UploadResponse` 并优先使用首批 URL。

## 8. 第二步：上传分片

分片编号从 1 开始：

```text
offset = (partNumber - 1) * partSize
size = min(partSize, fileSize - offset)
```

对每个分片执行：

```http
PUT uploadUrl
Content-Length: {size}

{part bytes}
```

注意：

1. `uploadUrl` 是对象存储预签名 URL，不是媒体服务接口。
2. PUT 成功通常返回 200 或 204。
3. PUT 失败、网络断开、URL 过期时，该 part 视为失败。
4. 下次恢复时重新调用创建/恢复接口，服务端会返回已上传 part，客户端只补传缺失 part。

Python 推荐使用 `os.pread` 读取分片，避免多线程共享文件游标：

```python
offset = (part_number - 1) * part_size
size = min(part_size, file_size - offset)
data = os.pread(fd, size, offset)
```

并发建议：

```text
RECORDING_UPLOAD_PART_CONCURRENCY=4
RECORDING_UPLOAD_PART_URL_BATCH_SIZE=16
```

这表示一次最多申请 16 个 part URL，但实际最多同时 PUT 4 个分片。

## 9. 需要更多 part URL 时

如果创建接口返回的 `partUrls` 不够，或者 URL 过期，调用：

```http
POST {MEDIA_SERVICE_URL}/api/media/files/multipart-uploads/{uploadId}/part-urls
Content-Type: application/json
X-Robot-Id: test111
```

请求体：

```json
{
  "partNumbers": [3, 4, 5, 6]
}
```

响应：

```json
{
  "expiresAt": "2026-07-22T15:41:17+08:00",
  "parts": [
    {
      "partNumber": 3,
      "uploadUrl": "http://minio-presigned-url-for-part-3"
    }
  ]
}
```

限制：

```text
MEDIA_FILE_MAX_PART_URLS_PER_REQUEST=16
```

一次请求不要超过服务端配置的最大数量。

## 10. 第三步：完成上传

所有缺失 part 都 PUT 成功后，必须调用 complete：

```http
POST {MEDIA_SERVICE_URL}/api/media/files/multipart-uploads/{uploadId}/complete
X-Robot-Id: test111
```

响应：

```json
{
  "fileId": "file_abc123",
  "status": "PROCESSING",
  "fileSize": 18268330,
  "ready": false,
  "errorCode": null,
  "errorMessage": null,
  "uploadedAt": "2026-07-22T15:28:00+08:00"
}
```

视频通常返回 `PROCESSING`。这表示原始 MP4 已上传完成，服务端正在做 HLS 处理。

如果服务端返回：

```text
UPLOAD_INCOMPLETE
UPLOAD_SIZE_MISMATCH
```

说明分片缺失、分片不连续或合并后的大小与 `fileSize` 不一致。客户端应重新 create_or_resume，检查缺失 part 后重传。

## 11. 第四步：轮询处理状态

```http
GET {MEDIA_SERVICE_URL}/api/media/files/{fileId}/status
X-Robot-Id: test111
```

响应：

```json
{
  "fileId": "file_abc123",
  "status": "READY",
  "fileSize": 18268330,
  "ready": true,
  "errorCode": null,
  "errorMessage": null,
  "uploadedAt": "2026-07-22T15:28:00+08:00"
}
```

状态处理：

| status | 含义 | 客户端动作 |
|---|---|---|
| `UPLOADING` | 原始文件仍在上传 | 正常不应在 complete 后长期停留 |
| `PROCESSING` | 原始文件已完成，服务端处理中 | 继续轮询 |
| `READY` | 文件可用，可播放/下载 | 保存 `fileId`，本地可进入保留/清理逻辑 |
| `FAILED` | 服务端处理失败 | 记录 `errorCode/errorMessage`，等待人工或重试策略 |

建议轮询间隔复用：

```text
RECORDING_UPLOAD_SCAN_INTERVAL_MS=30000
```

## 12. 本地 manifest

Python 客户端必须保存本地 manifest，避免重启后丢失上传状态。

默认路径：

```text
./recording-upload-manifest.json
```

建议保存字段：

```json
{
  "tasks": {
    "sourceFileId": {
      "sourceFileId": "test111/task-exec-20260722-001/camera01/visible_20260722_152451.mp4/18268330/1784705200",
      "filePath": "./recordings/visible_20260722_152451.mp4",
      "fileSize": 18268330,
      "createdAt": "2026-07-22T15:24:51Z",
      "fileId": "file_abc123",
      "uploadId": "upl_abc123",
      "status": "PROCESSING",
      "error": "",
      "updatedAt": "2026-07-22T15:28:00Z"
    }
  }
}
```

manifest 作用：

1. 避免重复发现同一个本地文件。
2. 重启后继续上传同一个 `sourceFileId`。
3. complete 后继续用 `fileId` 轮询状态。
4. READY 后本地文件被清理，也能保留 `LOCAL_DELETED` 状态。

## 13. 常见错误

### 13.1 429 Too Many Requests

原因：

```text
同一 robotId 的 ACTIVE 上传会话超过限制
```

默认：

```text
MEDIA_FILE_MAX_ACTIVE_UPLOADS_PER_ROBOT=2
MEDIA_FILE_MAX_ACTIVE_UPLOADS_GLOBAL=50
```

常见触发方式：

1. 没有传 `sourceFileId`，重复 create 产生多个 ACTIVE 上传。
2. create 后没有继续 PUT 和 complete。
3. 客户端崩溃留下 ACTIVE 上传，会等过期清理。

处理：

1. 必须传稳定 `sourceFileId`。
2. 客户端限制多文件并发，默认 2。
3. 对 429 做退避重试。
4. 排查数据库 `media_file_upload.status=ACTIVE` 的记录。

### 13.2 409 INVALID_STATE：准备文件存储桶失败

原因：

```text
媒体服务访问 MinIO/S3 失败，bucketExists 或 makeBucket 失败
```

检查：

```text
MINIO_ENDPOINT
MINIO_ACCESS_KEY
MINIO_SECRET_KEY
MINIO_BUCKET
MINIO_ENABLED
```

### 13.3 409 SOURCE_FILE_CHANGED

原因：

```text
相同 sourceFileId 对应的 fileName/fileSize/fileType 发生变化
```

处理：

1. 如果是同一个文件，保持文件名、大小和类型一致。
2. 如果文件内容已经变化，生成新的 `sourceFileId`。

### 13.4 409 UPLOAD_NOT_ACTIVE

原因：

```text
uploadId 已完成、失败、过期，或不再是 ACTIVE
```

处理：

```text
重新调用 create_or_resume，根据 sourceFileId 恢复或创建新上传会话
```

### 13.5 403 UNTRUSTED_ROBOT_NETWORK

如果服务端启用了内网可信接入：

```text
MEDIA_FILE_TRUSTED_ROBOT_NETWORK_ENABLED=true
```

机器人请求来源 IP 必须命中：

```text
MEDIA_FILE_TRUSTED_ROBOT_CIDRS
```

否则会被拦截。

## 14. Python 客户端建议改动点

当前 `python-client/robot_media_client/recordingupload/client.py` 的 `create_or_resume()` 只发送：

```text
sourceFileId
deviceId
fileType
fileName
contentType
fileSize
```

任务视频对接建议补充：

```text
robotId
extensionId
metadata
```

建议签名调整为：

```python
def create_or_resume(
    self,
    source_file_id: str,
    device_id: str,
    file_type: str,
    file_name: str,
    content_type: str,
    file_size: int,
    metadata: str = "",
    extension_id: str = "",
) -> UploadResponse:
    ...
```

请求体增加：

```python
{
    "robotId": self.robot_id,
    "deviceId": device_id,
    "extensionId": extension_id or None,
    "sourceFileId": source_file_id,
    "fileType": file_type,
    "fileName": file_name,
    "contentType": content_type,
    "fileSize": file_size,
    "metadata": metadata or None,
}
```

同时建议 `UploadResponse` 增加 `part_urls` 字段，用于复用创建接口返回的首批 URL，减少一次 `/part-urls` 请求。

## 15. 最小 Python 伪代码

```python
import json
import os
from pathlib import Path

path = Path("./recordings/visible_20260722_152451.mp4")
stat = path.stat()

robot_id = "test111"
device_id = "camera01"
source_file_id = f"{robot_id}/task-exec-20260722-001/{device_id}/{path.name}/{stat.st_size}/{int(stat.st_mtime)}"
metadata = json.dumps({
    "type": "task_recording",
    "cameraId": device_id,
    "channel": "visible",
    "taskExecutionId": "task-exec-20260722-001",
}, ensure_ascii=False)

upload = client.create_or_resume(
    source_file_id=source_file_id,
    device_id=device_id,
    file_type="VIDEO",
    file_name=path.name,
    content_type="video/mp4",
    file_size=stat.st_size,
    metadata=metadata,
)

if upload.upload_id:
    with open(path, "rb") as file:
        upload_missing_parts(
            client=client,
            file=file,
            file_size=stat.st_size,
            upload=upload,
            concurrency=4,
            url_batch_size=16,
        )
    status = client.complete(upload.upload_id)
else:
    status = client.status(upload.file_id)

while status.status == "PROCESSING":
    time.sleep(30)
    status = client.status(upload.file_id)

if status.status != "READY":
    raise RuntimeError(f"video processing failed: {status.error_code}")
```

## 16. 对接检查清单

上线前逐项确认：

1. `MEDIA_SERVICE_URL` 使用 HTTP/HTTPS 正确协议和端口。
2. 每个任务视频都有稳定 `sourceFileId`。
3. `metadata` 能区分 `task_recording`、摄像头、任务执行 ID。
4. create 返回 429 时做退避重试，不要高速重试。
5. PUT 分片失败时重新申请该 part URL。
6. 所有 part 成功后一定调用 complete。
7. complete 后继续轮询到 READY。
8. 本地 manifest 持久化 `fileId/uploadId/status/error`。
9. 不上传正在写入的 MP4，推荐 `.tmp` 写完后 rename。
10. 同一机器人并发上传数不超过服务端配置。
